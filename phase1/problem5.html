#5 — Merging Two Sorted Delivery Lists

🔖 Pattern: Two Pointer (Sorted Merge)

🎯 Real-World Scenario START HERE ▼

Tum ek delivery company mein kaam karte ho. Do warehouses se sorted delivery schedules aaye hain:

  • Warehouse A (by time): [9:00, 11:00, 14:00, 17:00]
  • Warehouse B (by time): [8:30, 10:00, 13:00, 15:30, 18:00]

Boss bola: "Dono lists ko merge karke ek single sorted timeline banao."

Result: [8:30, 9:00, 10:00, 11:00, 13:00, 14:00, 15:30, 17:00, 18:00]

In numbers: merge [1, 3, 5, 7] and [2, 4, 6, 8, 10] → [1, 2, 3, 4, 5, 6, 7, 8, 10]

💡 Dono already sorted hain — kya hum bina re-sorting ke merge kar sakte hain?

🧠 1. Problem Understanding STEP 1 ▼

What exactly is being asked?

  • Given two sorted arrays, merge them into one sorted array
  • The result must also be sorted

Input → Output:

  • Input: int[] a (sorted), int[] b (sorted)
  • Output: int[] (merged, sorted)

Key observation:

  • Both arrays are ALREADY sorted — this is a huge advantage
  • We don't need a general sorting algorithm (O(n log n))
  • We can exploit the sorted property to merge in O(n + m)

Connection to previous problems:

  • Problem #4 ke reverse() mein humne two pointers use kiye (left & right moving inward)
  • Ab hum two pointers ka proper introduction karenge — but this time two separate pointers on two separate arrays, both moving forward
💪 2. Brute Force Thinking STEP 2 ▼

First instinct: Concatenate both arrays, then sort.

// Brute Force — O((n+m) log(n+m))
int[] result = new int[a.length + b.length];
System.arraycopy(a, 0, result, 0, a.length);
System.arraycopy(b, 0, result, a.length, b.length);
Arrays.sort(result);
return result;

Problem:

  • Time: O((n+m) log(n+m)) — we're throwing away the "already sorted" information
  • Sorting from scratch when data is already sorted = wasted work

🤔 Dono sorted hain toh smallest element dono ke front mein se kisi ek pe hoga. Compare karo, chhota wala utha lo. Repeat.

⚡ 3. Optimization Thinking STEP 3 ▼

Key Insight — The Card Deck Analogy:

Imagine two sorted stacks of cards face-up on a table. To merge them:

  1. Look at the top card of both stacks
  2. Pick the smaller one, place it in the result pile
  3. Repeat until both stacks are empty

You never need to go back or re-sort. Each card is picked exactly once.

Translated to code:

  • Pointer i on array a, pointer j on array b
  • Compare a[i] vs b[j]
  • Smaller one goes into result, advance that pointer
  • When one array is exhausted, copy the rest of the other

💡 Pattern identified: Two Pointer — one pointer per sorted input, both moving forward

🎯 4. Final Approach (Pattern-Based) STEP 4 ▼

Pattern Name: Two Pointer (Sorted Merge)

Why this pattern fits:

  • Both inputs are sorted — we can make progress by always picking the smaller front element
  • Each pointer only moves forward — no backtracking needed
  • Total work = n + m (each element touched exactly once)

Algorithm:

  1. Create result array of size n + m
  2. Initialize i = 0, j = 0, k = 0
  3. While both i < n and j < m:
    • If a[i] <= b[j]: result[k++] = a[i++]
    • Else: result[k++] = b[j++]
  4. Copy remaining elements from whichever array isn't exhausted

💡 Using <= (not <) ensures stability — equal elements from a come before those from b.

⚙️ 6. Internal Mechanics STEP 6 ▼

Why two pointers work on sorted data:

  • Sorted = smallest is always at the front of what's remaining
  • So the global smallest among all remaining elements is either a[i] or b[j]
  • Pick the smaller → it's guaranteed to be the next element in the merged result
  • Advance that pointer → the new front is the next smallest from that array

Why we never go back:

  • Once we pick a[i], everything before i in a was already picked (smaller)
  • Everything after i in a is larger (sorted property)
  • So moving forward is always correct — no need to revisit

📊 7. Complexity Analysis STEP 7 ▼
Time: O(n + m)
Space: O(n + m)

Comparison:

Approach Time Uses sorted property?
Concat + SortO((n+m) log(n+m))No — wastes info

Space is O(n+m) for the result array — unavoidable since we need to store the merged output.

☕ 8. Java Implementation STEP 8 ▼ 
import java.util.Arrays;

public class MergeSortedArrays {

    public static int[] merge(int[] a, int[] b) {
        int n = a.length, m = b.length;
        int[] result = new int[n + m];
        int i = 0, j = 0, k = 0;

        while (i < n && j < m) {
            if (a[i] <= b[j]) {
                result[k++] = a[i++];
            } else {
                result[k++] = b[j++];
            }
        }

        while (i < n) result[k++] = a[i++];
        while (j < m) result[k++] = b[j++];

        return result;
    }

    public static void main(String[] args) {
        int[] a = {1, 3, 5, 7};
        int[] b = {2, 4, 6, 8, 10};
        System.out.println(Arrays.toString(merge(a, b)));
        // [1, 2, 3, 4, 5, 6, 7, 8, 10]
    }
}

Variant: Merge Sorted Array In-Place (LeetCode #88)

Array a has extra space at the end to hold b. Merge without extra array.

💡 Trick: Start from the END (largest elements) and fill backwards. Avoids overwriting.

// a has size n+m, first n elements filled, last m are 0
public static void mergeInPlace(int[] a, int n, int[] b, int m) {
    int i = n - 1, j = m - 1, k = n + m - 1;

    while (i >= 0 && j >= 0) {
        if (a[i] >= b[j]) {
            a[k--] = a[i--];
        } else {
            a[k--] = b[j--];
        }
    }

    while (j >= 0) a[k--] = b[j--];
}

No need to copy remaining a elements — they're already in place.

⚠️ 10. Common Mistakes STEP 10 ▼
  1. Forgetting to copy remaining elements

    ❌ Only the main while loop — misses the tail of the longer array

    ✅ Always add two cleanup loops (or use System.arraycopy)

  2. Using < instead of <=

    Functionally both work for correctness, but <= ensures stability (equal elements from a come first). Important in real-world sorting.

  3. In-place variant: filling from the front

    ❌ Overwrites elements in a that haven't been processed yet

    ✅ Fill from the back (largest first) — safe because the back is empty space

  4. Assuming both arrays are same size

    ✅ Always handle different sizes — the cleanup loops take care of this

🔑 11. When to Use This Pattern STEP 11 ▼

Signals to पहचान (identify) this pattern:

  • ✅ "Merge two sorted ___" (arrays, lists, streams)
  • ✅ "Intersection of two sorted arrays"
  • ✅ "Union of two sorted arrays"
  • ✅ Input is sorted and you need to combine/compare two sequences
  • ✅ Merge step of Merge Sort (Problem #24 will use this exact logic)

This merge logic appears in:

  • Merge Sort algorithm (divide & conquer)
  • External sorting (merging sorted files from disk)
  • Database merge joins
  • Merging k sorted lists (using heap — advanced)
🚫 12. When NOT to Use STEP 12 ▼
  • ❌ Arrays are NOT sorted — two pointer merge won't produce sorted output
  • ❌ You need to merge and remove duplicates — need extra check (if result[k-1] != current)
  • ❌ Merging more than 2 arrays — use a min-heap for k-way merge (O(N log k))
  • ❌ Arrays are sorted in different orders (one ascending, one descending) — reverse one first
💡 "Sorted input = free information. Two pointers exploit it — compare fronts, pick smaller, move forward. Never look back."
