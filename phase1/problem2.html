#2 — Duplicate Customer Complaint

🔖 Pattern: Frequency Counting (HashMap)

🎯 Real-World Scenario START HERE ▼

Tum ek customer support manager ho. Tumhare paas ek list hai of complaint IDs jo aaj aaye:

[101, 203, 101, 305, 203, 101, 407]

Boss ne bola: "Bata, kaunsa complaint ID sabse zyada baar aaya? Aur kitni baar?"

Simpler version pehle: Kya is list mein koi duplicate hai?

Answer: Haan — 101 (3 times), 203 (2 times). Most frequent = 101.

🧠 1. Problem Understanding STEP 1 ▼

What exactly is being asked?

  • Given an array of integers, determine if any value appears more than once
  • Variation 1: Return true/false (contains duplicate)
  • Variation 2: Return the first duplicate found
  • Variation 3: Return the element with highest frequency

Input → Output Clarity:

  • Input: int[] nums
  • Output: boolean (for basic version) or int (for frequency version)

Connection to Problem #1:

  • Problem #1 mein humne ek single variable (minPrice) se past track kiya
  • Ab ek variable kaafi nahi — humein har element ka count track karna hai
  • Single variable → Map of variables (HashMap)
💪 2. Brute Force Thinking STEP 2 ▼

First instinct: For each element, check if it appears again later in the array.

// Brute Force — O(n²)
public static boolean containsDuplicate(int[] nums) {
    for (int i = 0; i < nums.length; i++) {
        for (int j = i + 1; j < nums.length; j++) {
            if (nums[i] == nums[j]) return true;
        }
    }
    return false;
}

Problem: Same old nested loop → O(n²).

Alternative brute force: Sort the array first, then check adjacent elements.

// Sort-based — O(n log n)
Arrays.sort(nums);
for (int i = 1; i < nums.length; i++) {
    if (nums[i] == nums[i - 1]) return true;
}
return false;

Better — O(n log n). But sorting modifies the original array. Can we do O(n)?

🤔 Sochne ka point: Kya hum "pehle dekha hai ya nahi" instantly check kar sakte hain?

⚡ 3. Optimization Thinking STEP 3 ▼

Key Insight: Humein ek "memory" chahiye jo bata sake — "yeh element pehle aa chuka hai ya nahi?"

Problem #1 mein ek variable kaafi tha kyunki sirf min track karna tha. Ab har element ka record chahiye.

What data structure gives O(1) lookup?

  • Array? — Only if values are small and non-negative (index-based)
  • HashSet — O(1) average lookup for "seen or not seen"
  • HashMap — O(1) average lookup + stores count (frequency)

Trade-off:

Approach Time Space Modifies Input?
Nested LoopO(n²)O(1)No
SortingO(n log n)O(1)Yes
HashSet/HashMapO(n)O(n)No

💡 Pattern identified: Frequency Counting with HashMap — trade space for time.

🎯 4. Final Approach (Pattern-Based) STEP 4 ▼

Pattern Name: Frequency Counting (HashMap / HashSet)

Why this pattern fits:

  • We need to "remember" what we've seen before — that's a lookup problem
  • HashMap gives O(1) insert + O(1) lookup
  • We're trading O(n) space to eliminate the inner loop entirely

Two variants:

  • HashSet — when you only need "seen or not" (contains duplicate)
  • HashMap — when you need the count/frequency (most frequent element)

Algorithm (HashSet — duplicate check):

  1. Create an empty HashSet
  2. For each element in array:
    • If element already in set → return true (duplicate found)
    • Else → add element to set
  3. If loop ends → return false (no duplicates)

Algorithm (HashMap — frequency count):

  1. Create an empty HashMap<Integer, Integer>
  2. For each element: map.put(num, map.getOrDefault(num, 0) + 1)
  3. Iterate map to find element with highest count

⚙️ 6. Internal Mechanics STEP 6 ▼

How HashMap works internally (simplified):

  • HashMap internally ek array of "buckets" maintain karta hai
  • Jab tum put(key, value) karte ho:
    • Key ka hashCode() calculate hota hai
    • hashCode % arraySize → bucket index milta hai
    • Us bucket mein key-value pair store hota hai
  • Jab tum get(key) karte ho → same hash → same bucket → O(1) lookup

Why O(1) average?

  • Hash function distributes keys evenly across buckets
  • Worst case (all keys same bucket) → O(n), but this is extremely rare
  • Java 8+ uses balanced trees inside buckets when collisions exceed 8 → worst case O(log n)

HashSet vs HashMap:

  • HashSet internally IS a HashMap — it stores keys with a dummy value
  • set.add(x) = map.put(x, DUMMY)
  • Use HashSet when you only care about presence, HashMap when you need associated data (count, index, etc.)
📊 7. Complexity Analysis STEP 7 ▼

HashSet approach (contains duplicate):

Time: O(n)
Space: O(n)

HashMap approach (frequency count):

Time: O(n)
Space: O(n)

Space-time trade-off: We use O(n) extra memory to bring time from O(n²) down to O(n). Almost always worth it.

☕ 8. Java Implementation STEP 8 ▼

Variant 1: Contains Duplicate (HashSet)

import java.util.HashSet;

public class DuplicateCheck {

    public static boolean containsDuplicate(int[] nums) {
        HashSet<Integer> seen = new HashSet<>();
        for (int num : nums) {
            if (!seen.add(num)) return true;
        }
        return false;
    }

    public static void main(String[] args) {
        int[] nums = {101, 203, 101, 305};
        System.out.println(containsDuplicate(nums)); // true
    }
}

💡 Trick: set.add() returns false if element already exists. So !seen.add(num) is a one-liner duplicate check.

Variant 2: Most Frequent Element (HashMap)

import java.util.HashMap;
import java.util.Map;

public class MostFrequent {

    public static int mostFrequent(int[] nums) {
        HashMap<Integer, Integer> freq = new HashMap<>();

        for (int num : nums) {
            freq.put(num, freq.getOrDefault(num, 0) + 1);
        }

        int maxCount = 0, result = nums[0];
        for (Map.Entry<Integer, Integer> entry : freq.entrySet()) {
            if (entry.getValue() > maxCount) {
                maxCount = entry.getValue();
                result = entry.getKey();
            }
        }
        return result;
    }

    public static void main(String[] args) {
        int[] complaints = {101, 203, 101, 305, 203, 101, 407};
        System.out.println("Most frequent: " + mostFrequent(complaints)); // 101
    }
}

⚠️ 10. Common Mistakes STEP 10 ▼
  1. Using == with Integer objects instead of .equals()

    ❌ For Integer values > 127, == compares references, not values

    ✅ Use int primitives or .equals() for Integer objects

    Our code uses int[] so autoboxing handles this, but be aware when using Integer[]

  2. Forgetting getOrDefault

    ❌ map.get(num) + 1 → NullPointerException if key doesn't exist

    ✅ map.getOrDefault(num, 0) + 1 → safe, returns 0 if absent

  3. Using HashMap when HashSet is enough

    If you only need "exists or not" → HashSet is cleaner and more readable

    HashMap is for when you need the count or associated data

  4. Not considering the set.add() return value trick

    ✅ set.add(x) returns false if x already exists — one line does both check + insert

🔑 11. When to Use This Pattern STEP 11 ▼

Signals to पहचान (identify) this pattern:

  • ✅ "Find duplicates" / "Find frequency" / "Count occurrences"
  • ✅ "Have we seen this before?" type questions
  • ✅ "Two Sum" — find two numbers that add to target (HashMap stores complement)
  • ✅ "Group by" — group anagrams, group by frequency
  • ✅ Any problem where O(1) lookup would eliminate an inner loop

Real-world uses:

  • Spam detection (seen this message before?)
  • Vote counting / election results
  • Web analytics (page visit frequency)
  • Caching (is this result already computed?)
  • Database indexing (hash indexes)
🚫 12. When NOT to Use STEP 12 ▼
  • ❌ When space is strictly O(1) — sorting might be the only option
  • ❌ When you need elements in sorted order — HashMap doesn't maintain order (use TreeMap)
  • ❌ When the range of values is small and known — a simple array is faster than HashMap
  • ❌ When you need "k-th most frequent" — HashMap alone isn't enough (need heap or bucket sort on top)
