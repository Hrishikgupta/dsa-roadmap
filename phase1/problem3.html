#3 — Balancing the Warehouse Stock

🔖 Pattern: Prefix Sum

🎯 Real-World Scenario START HERE ▼

Tum ek warehouse manager ho. Tumhare paas 7 days ka stock data hai:

[3, 1, 4, 1, 5, 9, 2] (units received each day)

Boss baar baar poochta hai:

  • "Day 2 se Day 5 tak total kitna stock aaya?" → 1+4+1+5 = 11
  • "Day 0 se Day 3 tak?" → 3+1+4+1 = 9
  • "Day 4 se Day 6 tak?" → 5+9+2 = 16

Har baar loop lagake sum karna padega? Agar boss 10,000 baar pooche toh?

💡 Kya hum ek baar pre-compute karke har query O(1) mein answer kar sakte hain?

🧠 1. Problem Understanding STEP 1 ▼

What exactly is being asked?

  • Given an array, answer multiple "range sum queries" efficiently
  • Each query: given left and right index, return sum of elements from left to right (inclusive)

Input → Output Clarity:

  • Input: int[] nums + multiple queries (left, right)
  • Output: int sum for each query

Connection to previous problems:

  • Problem #1: Track one value (minPrice) → avoid inner loop
  • Problem #2: Track multiple values (HashMap) → avoid inner loop
  • Problem #3: Pre-compute cumulative data → avoid repeated work across multiple queries
💪 2. Brute Force Thinking STEP 2 ▼

First instinct: For each query, loop from left to right and add up.

// Brute Force — O(n) per query, O(n × q) total
public int rangeSum(int[] nums, int left, int right) {
    int sum = 0;
    for (int i = left; i <= right; i++) {
        sum += nums[i];
    }
    return sum;
}

Problem:

  • Each query takes O(n) in worst case
  • If there are q queries → total = O(n × q)
  • For n = 100,000 and q = 100,000 → 10 billion operations. TLE.

🤔 Sochne ka point: Hum baar baar same elements add kar rahe hain. Kya yeh kaam pehle se kar sakte hain?

⚡ 3. Optimization Thinking STEP 3 ▼

Key Insight: Agar tumhare paas "day 0 se day i tak ka total" pehle se stored ho, toh kisi bhi range ka sum ek subtraction mein mil jayega.

Analogy — Odometer (car ka meter):

  • Tumhari car ka odometer 500 km pe hai jab tum Delhi se nikle
  • Jaipur pahunchte ho toh odometer 780 km dikhata hai
  • Delhi → Jaipur distance = 780 - 500 = 280 km
  • Tumne beech ka har km count nahi kiya — sirf do readings ka difference liya

Same idea with arrays:

  • Build a "cumulative sum" array: prefix[i] = sum of nums[0..i]
  • Range sum (left, right) = prefix[right] - prefix[left - 1]
  • One subtraction instead of a loop → O(1) per query

💡 Pattern identified: Prefix Sum — pre-compute to answer queries in O(1)

🎯 4. Final Approach (Pattern-Based) STEP 4 ▼

Pattern Name: Prefix Sum

Why this pattern fits:

  • Multiple queries on the same static array
  • Each query asks for a "range aggregate" (sum, in this case)
  • Pre-computation cost is O(n) once, then every query is O(1)

Algorithm:

  1. Create prefix[] array of size n + 1 (extra slot to avoid boundary checks)
  2. prefix[0] = 0 (sum of zero elements)
  3. For i = 0 to n-1: prefix[i+1] = prefix[i] + nums[i]
  4. For query (left, right): answer = prefix[right+1] - prefix[left]

💡 Size n+1 trick: prefix[0] = 0 means we never need to special-case left == 0. Clean formula works for all ranges.

⚙️ 6. Internal Mechanics STEP 6 ▼

Why does subtraction work?

  • prefix[right+1] = sum of nums[0] + nums[1] + ... + nums[right]
  • prefix[left] = sum of nums[0] + nums[1] + ... + nums[left-1]
  • Subtract → the overlapping part (0 to left-1) cancels out
  • What remains = nums[left] + nums[left+1] + ... + nums[right] ✓

Visual:

// prefix[right+1] covers this:
[  3  1  4  1  5  9  2  ]
 ←————————————————→

// prefix[left] covers this:
[  3  1  |              ]
 ←——→

// Difference = the green part only
         4  1  5  9  = 19

Why size n+1?

  • If we used size n, then for query (0, right) we'd need prefix[-1] which doesn't exist
  • Size n+1 with prefix[0] = 0 means the formula prefix[right+1] - prefix[left] works universally
  • No special cases, no if-else — clean math
📊 7. Complexity Analysis STEP 7 ▼

Pre-computation (building prefix array):

Time: O(n)
Space: O(n)

Each query:

Time: O(1)

Total for q queries:

Approach Total Time n=100K, q=100K
Brute ForceO(n × q)10,000,000,000
Prefix SumO(n + q)200,000

50,000x faster. Pre-computation is a superpower.

☕ 8. Java Implementation STEP 8 ▼ 
public class PrefixSum {

    private int[] prefix;

    public PrefixSum(int[] nums) {
        prefix = new int[nums.length + 1];
        for (int i = 0; i < nums.length; i++) {
            prefix[i + 1] = prefix[i] + nums[i];
        }
    }

    public int rangeSum(int left, int right) {
        return prefix[right + 1] - prefix[left];
    }

    public static void main(String[] args) {
        int[] stock = {3, 1, 4, 1, 5, 9, 2};
        PrefixSum ps = new PrefixSum(stock);

        System.out.println(ps.rangeSum(2, 5)); // 19
        System.out.println(ps.rangeSum(0, 3)); // 9
        System.out.println(ps.rangeSum(4, 6)); // 16
    }
}

💡 Constructor does O(n) work once. Every rangeSum() call is O(1). This is the LeetCode "NumArray" problem (LC #303).

⚠️ 10. Common Mistakes STEP 10 ▼
  1. Off-by-one error in the formula

    ❌ prefix[right] - prefix[left] → misses nums[right]

    ✅ prefix[right+1] - prefix[left] → includes both endpoints

    Dry run with a small example if you're ever confused about the +1

  2. Using size n instead of n+1

    ❌ Then you need special handling for left == 0

    ✅ Size n+1 with prefix[0] = 0 eliminates all edge cases

  3. Forgetting that prefix sum is for STATIC arrays only

    ❌ If array changes after building prefix → all answers become wrong

    For dynamic arrays (with updates), you need a Segment Tree or BIT (advanced topic)

  4. Integer overflow

    ❌ If elements are large, cumulative sum can overflow int

    ✅ Use long[] for prefix array when values are large

🔑 11. When to Use This Pattern STEP 11 ▼

Signals to पहचान (identify) this pattern:

  • ✅ "Range sum query" / "Subarray sum"
  • ✅ Multiple queries on the same array
  • ✅ "Sum of elements between index i and j"
  • ✅ "Number of subarrays with sum equal to K" (prefix sum + HashMap combo — advanced)
  • ✅ 2D version: "Sum of submatrix" (2D prefix sum)

Variations you'll see:

  • Prefix XOR — for range XOR queries
  • Prefix product — for range product (careful with zeros)
  • Difference array — inverse of prefix sum (for range updates)
  • 2D prefix sum — for matrix region queries
🚫 12. When NOT to Use STEP 12 ▼
  • ❌ Array is being modified between queries (need Segment Tree / BIT)
  • ❌ Only one query — just loop once, no need to pre-compute
  • ❌ Range min/max queries — prefix sum doesn't help (need Sparse Table / Segment Tree)
  • ❌ Non-associative operations — prefix trick only works for operations where order doesn't matter (sum, XOR, product)
💡 "Jab same array pe baar baar range sum poochha jaaye — pehle prefix bana lo, phir har query O(1) mein answer karo."
