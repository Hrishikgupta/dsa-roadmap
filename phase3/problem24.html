#24 — Merge Sort — Divide the Army, Conquer the War

🔖 Pattern: Divide and Conquer

🎯 Real-World Scenario START HERE ▼

Tum ek general ho. Tumhari army mein 8 soldiers hain, unordered:

[38, 27, 43, 3, 9, 82, 10, 76]

Tumhe unhe rank order mein line up karna hai. Strategy:

  1. Divide: Army ko do halves mein split karo
  2. Conquer: Har half ko apne lieutenant ko de do — "sort karo" (recursion!)
  3. Merge: Dono sorted halves ko merge karo (Problem #5 ka two-pointer merge!)

Result: [3, 9, 10, 27, 38, 43, 76, 82]

Merge Sort — the most important O(n log n) sorting algorithm. Foundation for many advanced problems.

🧠 1. Problem Understanding STEP 1 ▼

What is Divide and Conquer?

  1. Divide: Split the problem into smaller subproblems
  2. Conquer: Solve each subproblem recursively
  3. Combine: Merge the solutions of subproblems into the final answer

How Merge Sort applies D&C:

  • Divide: split array into two halves at midpoint
  • Conquer: recursively sort each half
  • Combine: merge two sorted halves (this is Problem #5's two-pointer merge!)

Connection to previous problems:

From Concept Used Here

💡 Merge Sort = Recursion (#20) + Merge (#5). Two things you already know, combined into something powerful.

💪 2. Brute Force Thinking STEP 2 ▼

Simple sorting approaches:

  • Bubble Sort: O(n²) — compare adjacent, swap, repeat
  • Selection Sort: O(n²) — find min, place at front, repeat
  • Insertion Sort: O(n²) — insert each element into sorted portion

All O(n²). For n = 1,000,000 → 10¹² operations. Way too slow.

🤔 Can we do better? Yes — by dividing the problem. Sorting n elements is hard. Sorting n/2 elements is easier. Sorting 1 element is trivial. Then merge the sorted pieces.

⚡ 3. Optimization Thinking STEP 3 ▼

Key Insight — Why splitting helps:

  • Sorting n items: hard (O(n²) with simple methods)
  • Sorting n/2 items: 4× easier (n/2)² = n²/4
  • Sorting 1 item: trivial (already sorted!)
  • Merging two sorted halves: O(n) — we proved this in Problem #5

The math — why O(n log n):

Level 0: 1 array of size n         → merge cost: n
Level 1: 2 arrays of size n/2      → merge cost: n
Level 2: 4 arrays of size n/4      → merge cost: n
...                                              
Level k: n arrays of size 1        → merge cost: n

How many levels? log₂(n)
Cost per level: O(n)
Total: O(n) × O(log n) = O(n log n)

💡 Pattern: Divide and Conquer — split in half, solve recursively, merge in O(n) → total O(n log n)

🎯 4. Final Approach (Pattern-Based) STEP 4 ▼

Algorithm:

  1. Base case: array of size ≤ 1 → already sorted, return
  2. Divide: find mid = (left + right) / 2
  3. Conquer:
    • mergeSort(arr, left, mid)
    • mergeSort(arr, mid+1, right)
  4. Combine: merge the two sorted halves into one sorted array

The merge step (same as Problem #5):

  • Two pointers: one on each sorted half
  • Compare, pick smaller, advance that pointer
  • Copy remaining elements
  • Copy merged result back to original array
🔍 5. Step-by-Step Dry Run STEP 5 ▼

Input: [38, 27, 43, 3, 9, 82, 10, 76]

DIVIDE phase (top-down):

[38, 27, 43, 3, 9, 82, 10, 76]
         /                \
[38, 27, 43, 3]    [9, 82, 10, 76]
    /      \           /       \
[38, 27]  [43, 3]  [9, 82]  [10, 76]
  / \      / \      / \       / \
[38][27] [43][3]  [9][82]  [10][76]

MERGE phase (bottom-up):

[38][27] → [27, 38]    [43][3] → [3, 43]
[9][82] → [9, 82]     [10][76] → [10, 76]

[27, 38] + [3, 43] → [3, 27, 38, 43]
[9, 82] + [10, 76] → [9, 10, 76, 82]

[3, 27, 38, 43] + [9, 10, 76, 82]
→ [3, 9, 10, 27, 38, 43, 76, 82] ✓

3 levels of splitting (log₂8 = 3). Each level does O(n) merge work. Total: 3 × 8 = 24 operations ≈ O(n log n).

⚙️ 6. Internal Mechanics STEP 6 ▼

Merge Sort vs Quick Sort:

Aspect Merge Sort Quick Sort
DivideSplit at midpoint (easy)Partition around pivot (hard)
CombineMerge two sorted halves (hard)Nothing (elements already in place)
Worst caseO(n log n) alwaysO(n²) (bad pivot)
SpaceO(n) extraO(log n) (in-place)
Stable?YesNo
Best forLinked lists, external sort, stability neededArrays, in-memory, average case

Why Merge Sort matters beyond sorting:

  • The merge step can be modified to extract information DURING the merge
  • Problem #25 (next): Count Inversions — count how many pairs are out of order, using modified merge
  • This "extract info during merge" technique is a powerful interview pattern

Stability explained:

  • Stable sort: equal elements maintain their original relative order
  • Merge Sort is stable because during merge, when elements are equal, we pick from the LEFT half first
  • This matters when sorting objects by multiple keys (sort by name, then by age — stable sort preserves name order within same age)
📊 7. Complexity Analysis STEP 7 ▼
Time: O(n log n)
Space: O(n)

Recurrence: T(n) = 2·T(n/2) + O(n) → O(n log n) by Master Theorem.

Space: O(n) for the temporary merge array + O(log n) recursion stack.

Algorithm Time n = 1,000,000
Bubble/Selection/InsertionO(n²)10¹²
Merge SortO(n log n)~20,000,000

50,000× faster for 1 million elements. That's the power of divide and conquer.

☕ 8. Java Implementation STEP 8 ▼ 
import java.util.Arrays;

public class MergeSort {

    public static void mergeSort(int[] arr, int left, int right) {
        if (left >= right) return; // base case: 0 or 1 element

        int mid = left + (right - left) / 2;

        mergeSort(arr, left, mid);       // sort left half
        mergeSort(arr, mid + 1, right);  // sort right half
        merge(arr, left, mid, right);    // merge sorted halves
    }

    private static void merge(int[] arr, int left, int mid, int right) {
        int[] temp = new int[right - left + 1];
        int i = left, j = mid + 1, k = 0;

        while (i <= mid && j <= right) {
            if (arr[i] <= arr[j]) {
                temp[k++] = arr[i++];
            } else {
                temp[k++] = arr[j++];
            }
        }

        while (i <= mid) temp[k++] = arr[i++];
        while (j <= right) temp[k++] = arr[j++];

        System.arraycopy(temp, 0, arr, left, temp.length);
    }

    public static void main(String[] args) {
        int[] arr = {38, 27, 43, 3, 9, 82, 10, 76};
        mergeSort(arr, 0, arr.length - 1);
        System.out.println(Arrays.toString(arr));
        // [3, 9, 10, 27, 38, 43, 76, 82]
    }
}

💡 left + (right - left) / 2 instead of (left + right) / 2 — avoids integer overflow when left + right > Integer.MAX_VALUE.

⚠️ 10. Common Mistakes STEP 10 ▼
  1. Integer overflow in mid calculation

    ❌ (left + right) / 2 — overflows when left + right > 2³¹

    ✅ left + (right - left) / 2 — safe

  2. Not copying merged result back to original array

    ❌ Merge into temp but forget System.arraycopy back

    ✅ Always copy temp back: System.arraycopy(temp, 0, arr, left, temp.length)

  3. Wrong boundary in recursive calls

    ❌ mergeSort(arr, left, mid-1) and mergeSort(arr, mid, right) — can cause infinite recursion when left==right

    ✅ mergeSort(arr, left, mid) and mergeSort(arr, mid+1, right)

  4. Using < instead of <= in merge for stability

    ✅ arr[i] <= arr[j] — pick from left half when equal → maintains stability

🔑 11. When to Use This Pattern STEP 11 ▼

Signals to पहचान (identify):

  • ✅ "Sort an array" — when stability or guaranteed O(n log n) is needed
  • ✅ "Count inversions" / "Count something during merge" (Problem #25)
  • ✅ Sorting linked lists — merge sort is ideal (no random access needed)
  • ✅ External sorting — data too large for memory, sort chunks and merge
  • ✅ Any problem where "split, solve halves, combine" structure fits

Divide and Conquer family:

  • Merge Sort — split at mid, merge sorted halves
  • Quick Sort — partition around pivot, sort partitions
  • Binary Search — eliminate half each step
  • Closest pair of points — geometric D&C
  • Strassen's matrix multiplication — D&C on matrices
🚫 12. When NOT to Use STEP 12 ▼
  • ❌ Space is critical — merge sort needs O(n) extra. Use Quick Sort or Heap Sort for O(1) extra.
  • ❌ Nearly sorted data — Insertion Sort is O(n) on nearly sorted. Merge Sort is still O(n log n).
  • ❌ Small arrays (n < 20) — overhead of recursion. Use Insertion Sort. (Java's Arrays.sort does this hybrid.)
💡 "Divide and Conquer: split the problem, trust recursion to solve halves, merge the results. Merge Sort = O(n) merge × O(log n) levels = O(n log n)."
