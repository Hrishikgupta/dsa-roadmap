#23 — Phone Keypad Letter Combinations

🔖 Pattern: Recursion (Multi-branch)

🎯 Real-World Scenario START HERE ▼

Yaad hai purane Nokia phones? Har digit pe 3-4 letters mapped the:

2 → a b c       5 → j k l       8 → t u v
3 → d e f       6 → m n o       9 → w x y z
4 → g h i       7 → p q r s

User ne type kiya: "23"

Phone ko suggest karna hai: saare possible letter combinations

2 → {a,b,c}, 3 → {d,e,f}

This is LeetCode #17 — Letter Combinations of a Phone Number. Classic interview question at Google, Amazon, Meta.

🧠 1. Problem Understanding STEP 1 ▼

What exactly is being asked?

  • Given a string of digits (2-9), return all possible letter combinations
  • Each digit maps to a set of letters (like old phone keypad)
  • Order of combinations doesn't matter

Input → Output:

  • Input: String digits (e.g., "23")
  • Output: List<String> — all letter combinations

How this differs from #21 and #22:

# Problem Choices per step Total results
21SubsetsAlways 2 (include/exclude)2ⁿ
22Permutationsn, n-1, n-2... (decreasing)n!
23Phone keypad3 or 4 (varies per digit)3ᵏ to 4ᵏ

💡 This is the most general form of recursion: at each step, the number of choices depends on the input at that step.

💪 2. Brute Force Thinking STEP 2 ▼

Iterative approach (queue-based BFS):

// Start with [""]
// For each digit, expand every existing string with each letter
queue = [""]
for digit in digits:
    letters = mapping[digit]
    newQueue = []
    for combo in queue:
        for letter in letters:
            newQueue.add(combo + letter)
    queue = newQueue

Works, but creates many intermediate strings. Recursive approach is cleaner and more natural for this tree-shaped problem.

⚡ 3. Optimization Thinking STEP 3 ▼

Key Insight — The Combination Lock Analogy:

Imagine a lock with multiple dials. Each dial has different options:

  • Dial 1 (digit "2"): can be a, b, or c
  • Dial 2 (digit "3"): can be d, e, or f
  • Every combination of dial settings = one valid output

Recursively: fix dial 1 to 'a', then try all options for remaining dials. Then fix dial 1 to 'b', try all remaining. Then 'c'...

Recursive structure:

combine(index, currentString):
    if index == digits.length:    // all dials set
        add currentString to result
        return

    letters = mapping[digits[index]]  // options for this dial
    for letter in letters:
        combine(index + 1, currentString + letter)

No explicit backtracking needed here — string concatenation creates a new string each time (immutable in Java). The "undo" happens naturally when the loop moves to the next letter.

💡 Pattern: Multi-branch recursion — at each step, branch into all options for that step's input

🎯 4. Final Approach (Pattern-Based) STEP 4 ▼

Pattern: Multi-branch Recursion (Cartesian Product)

Algorithm:

  1. Create digit → letters mapping
  2. If digits is empty → return empty list
  3. Call combine(0, "")
  4. At each index:
    • Get letters for digits.charAt(index)
    • For each letter: recurse with index + 1 and current + letter
  5. When index == digits.length() → add current to result

💡 Using StringBuilder instead of string concatenation is faster (avoids creating new string objects), but string concat is cleaner for interviews.

🔍 5. Step-by-Step Dry Run STEP 5 ▼

Input: digits = "23"

Mapping: 2→"abc", 3→"def"

Call tree:

combine(0, "")
├── letter 'a' → combine(1, "a")
│   ├── letter 'd' → combine(2, "ad") → BASE: "ad" ✓
│   ├── letter 'e' → combine(2, "ae") → BASE: "ae" ✓
│   └── letter 'f' → combine(2, "af") → BASE: "af" ✓
├── letter 'b' → combine(1, "b")
│   ├── letter 'd' → "bd" ✓
│   ├── letter 'e' → "be" ✓
│   └── letter 'f' → "bf" ✓
└── letter 'c' → combine(1, "c")
    ├── letter 'd' → "cd" ✓
    ├── letter 'e' → "ce" ✓
    └── letter 'f' → "cf" ✓

Result: ["ad","ae","af","bd","be","bf","cd","ce","cf"] — 9 combinations ✓

Level 0: 3 branches (a,b,c). Level 1: 3 branches each (d,e,f). Total leaves = 3 × 3 = 9.

Another example: digits = "79"

7→"pqrs" (4 letters), 9→"wxyz" (4 letters) → 4 × 4 = 16 combinations

⚙️ 6. Internal Mechanics STEP 6 ▼

Why no explicit backtracking?

  • In #21 and #22, we modified a shared list (current.add() / current.remove())
  • Here, current + letter creates a NEW string — the old current is unchanged
  • When the recursive call returns, current is still the same → no undo needed
  • This is "implicit backtracking" via immutable strings

If using StringBuilder (mutable), you WOULD need explicit backtrack: sb.append(letter) → recurse → sb.deleteCharAt(sb.length()-1)

This is a Cartesian Product:

  • Set A = {a,b,c}, Set B = {d,e,f}
  • A × B = {(a,d),(a,e),(a,f),(b,d),(b,e),(b,f),(c,d),(c,e),(c,f)}
  • Recursion naturally computes the Cartesian product of multiple sets
  • For k digits with ~3 letters each: 3ᵏ results

The complete recursion pattern family:

Pattern Branching Results Backtrack?
Include/Exclude (#21)2 per step2ⁿYes (shared list)
Permutation (#22)n, n-1, n-2...n!Yes (used[] or swap)
Multi-branch (#23)Variable per step∏(choices)Implicit (new string)
📊 7. Complexity Analysis STEP 7 ▼
Time: O(4ᵏ × k)
Space: O(k)

Where k = number of digits. Worst case: all digits are 7 or 9 (4 letters each) → 4ᵏ combinations, each of length k.

Space: O(k) recursion depth. Output space O(4ᵏ × k) not counted.

☕ 8. Java Implementation STEP 8 ▼

String concatenation (clean, interview-friendly):

import java.util.*;

public class PhoneKeypad {

    private static final String[] MAP = {
        "", "", "abc", "def", "ghi", "jkl",
        "mno", "pqrs", "tuv", "wxyz"
    };

    public static List<String> letterCombinations(String digits) {
        List<String> result = new ArrayList<>();
        if (digits.isEmpty()) return result;
        combine(digits, 0, "", result);
        return result;
    }

    private static void combine(String digits, int idx,
                                  String current, List<String> result) {
        if (idx == digits.length()) {
            result.add(current);
            return;
        }

        String letters = MAP[digits.charAt(idx) - '0'];
        for (char letter : letters.toCharArray()) {
            combine(digits, idx + 1, current + letter, result);
        }
    }

    public static void main(String[] args) {
        System.out.println(letterCombinations("23"));
        // [ad, ae, af, bd, be, bf, cd, ce, cf]
    }
}

StringBuilder version (faster, explicit backtrack):

private static void combineSB(String digits, int idx,
                                StringBuilder sb, List<String> result) {
    if (idx == digits.length()) {
        result.add(sb.toString());
        return;
    }
    String letters = MAP[digits.charAt(idx) - '0'];
    for (char letter : letters.toCharArray()) {
        sb.append(letter);
        combineSB(digits, idx + 1, sb, result);
        sb.deleteCharAt(sb.length() - 1); // backtrack!
    }
}

💡 String concat: cleaner code, implicit backtrack. StringBuilder: faster, explicit backtrack. Both O(4ᵏ). Choose based on interview context.

⚠️ 10. Common Mistakes STEP 10 ▼
  1. Returning [""] for empty input instead of []

    ❌ Empty string in result is wrong — should be empty list

    ✅ Check digits.isEmpty() before starting recursion

  2. Wrong mapping index

    ❌ MAP[digits.charAt(idx)] — char '2' has ASCII value 50, not 2

    ✅ MAP[digits.charAt(idx) - '0'] — convert char to int

  3. Forgetting backtrack with StringBuilder

    String concat: no backtrack needed (immutable). StringBuilder: MUST deleteCharAt after recursion.

🔑 11. When to Use This Pattern STEP 11 ▼

Signals to पहचान (identify):

  • ✅ "All combinations from multiple groups" (Cartesian product)
  • ✅ Each position has its own set of options
  • ✅ "Letter combinations" / "Word generation from mapping"
  • ✅ Multiple independent choices that combine

Real-world applications:

  • T9 predictive text (old phones)
  • Password generation from character sets
  • Configuration combinations (size × color × material)
  • Test case generation (all input combinations)
🚫 12. When NOT to Use STEP 12 ▼
  • ❌ "Count combinations" without listing — use multiplication (3 × 3 = 9), not recursion
  • ❌ Very long digit string — 4¹⁰ = ~1M, 4²⁰ = ~1T. Exponential growth limits input size.
  • ❌ Need to find ONE specific combination — use targeted search, not generate all
💡 "Multiple groups of choices? Recurse through each group, branch into all options. It's a Cartesian product built by recursion."
