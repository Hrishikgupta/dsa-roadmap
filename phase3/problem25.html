#25 — Count Inversions in a Ranking List

🔖 Pattern: Modified Merge Sort

🎯 Real-World Scenario START HERE ▼

Tum ek movie recommendation engine bana rahe ho. Do users ke movie rankings compare karne hain:

  • User A ranks movies: [2, 4, 1, 3, 5] (movie 2 is #1 favorite, movie 4 is #2, etc.)
  • Sorted order would be: [1, 2, 3, 4, 5]

"Kitne pairs of movies are out of order?" — this measures how "unsorted" the list is.

An inversion = a pair (i, j) where i < j but arr[i] > arr[j].

  • (2,1): 2 > 1 ✓ inversion
  • (4,1): 4 > 1 ✓ inversion
  • (4,3): 4 > 3 ✓ inversion

Answer: 3 inversions

Classic interview problem. Used in collaborative filtering (recommendation systems) to measure similarity between rankings.

🧠 1. Problem Understanding STEP 1 ▼

What exactly is being asked?

  • Count the number of pairs (i, j) where i < j and arr[i] > arr[j]
  • Sorted array = 0 inversions. Reverse sorted = n(n-1)/2 inversions (maximum).

Input → Output:

  • Input: int[] arr
  • Output: long — count of inversions (can be very large, use long)

Connection to Problem #24:

  • Merge Sort sorts the array by fixing inversions
  • Every time we pick from the RIGHT half before the LEFT half during merge → that's an inversion!
  • We can COUNT inversions during the merge step without extra cost

💡 This is the "extract info during merge" technique — the merge step does double duty: sort + count.

💪 2. Brute Force Thinking STEP 2 ▼ 
// Brute Force — O(n²)
long count = 0;
for (int i = 0; i < n; i++) {
    for (int j = i + 1; j < n; j++) {
        if (arr[i] > arr[j]) count++;
    }
}
return count;

O(n²) — checks every pair. For n = 100,000 → 5 billion comparisons.

🤔 Merge Sort already compares elements during merge. Can we piggyback the inversion count onto that comparison?

⚡ 3. Optimization Thinking STEP 3 ▼

Key Insight — When does an inversion get counted during merge?

During merge of left half L and right half R:

  • If L[i] <= R[j] → no inversion, pick from L. Normal.
  • If L[i] > R[j] → inversion! Pick from R.

But how many inversions?

  • If L[i] > R[j], then ALL remaining elements in L (from i to end) are also > R[j]
  • Why? Because L is sorted! If L[i] > R[j], then L[i+1], L[i+2], ... are all ≥ L[i] > R[j]
  • So inversions contributed = mid - i + 1 (remaining elements in left half)
// During merge, when picking from right:
if (left[i] > right[j]) {
    count += (mid - i + 1);  // ALL remaining left elements form inversions with right[j]
    temp[k++] = right[j++];
}

💡 Pattern: Modified Merge Sort — add one line to the merge step to count inversions. Same O(n log n).

🎯 4. Final Approach (Pattern-Based) STEP 4 ▼

Algorithm: Exactly Merge Sort (#24), with one addition in the merge step.

  1. Split array in half, recursively count inversions in each half
  2. During merge: when left[i] > right[j], add (mid - i + 1) to count
  3. Total inversions = left inversions + right inversions + split inversions (from merge)

Three types of inversions:

  • Left inversions: both elements in left half → counted by recursive call on left
  • Right inversions: both elements in right half → counted by recursive call on right
  • Split inversions: one element in left, one in right → counted during merge
🔍 5. Step-by-Step Dry Run STEP 5 ▼

Input: [2, 4, 1, 3, 5]

Split:
[2, 4, 1, 3, 5]
    /         \
[2, 4]     [1, 3, 5]
 / \        /     \
[2][4]    [1]   [3, 5]
                 / \
               [3] [5]

Merge (bottom-up) with inversion counting:

Merge [3],[5] → [3,5]
  3 < 5 → pick 3 (no inversion). 5 remaining → pick 5.
  Split inversions: 0

Merge [1],[3,5] → [1,3,5]
  1 < 3 → pick 1 (no inversion). Copy [3,5].
  Split inversions: 0

Merge [2],[4] → [2,4]
  2 < 4 → pick 2 (no inversion). Copy [4].
  Split inversions: 0

Merge [2,4],[1,3,5] → [1,2,3,4,5]
  2 > 1 → pick 1. Inversions += (2-0) = 2 (both 2 and 4 > 1)
  2 < 3 → pick 2. No inversion.
  4 > 3 → pick 3. Inversions += (2-1) = 1 (only 4 > 3)
  4 < 5 → pick 4. No inversion.
  Copy [5].
  Split inversions: 2 + 1 = 3

Total: 0 + 0 + 0 + 0 + 3 = 3 inversions ✓

The key moment: when merging [2,4] with [1,3,5], picking 1 before 2 means BOTH 2 and 4 are inverted with 1 → count += 2 (remaining left elements).

⚙️ 6. Internal Mechanics STEP 6 ▼

Why mid - i + 1 works:

Left half (sorted): [2, 4, 7, 9]   i points to 4
Right half (sorted): [3, ...]       j points to 3

4 > 3 → inversion!
But also: 7 > 3 and 9 > 3 (everything after i in left)
Count = elements from i to end of left = mid - i + 1 = 3

This works because the left half is SORTED. If L[i] > R[j], then L[i], L[i+1], ..., L[mid] are ALL > R[j].

The "extract info during merge" technique:

Merge Sort's merge step compares elements from two sorted halves. You can piggyback ANY cross-half counting onto this comparison:

Problem What to count during merge
Count Inversions (this)When right picked before left → count remaining left
Reverse Pairs (LC #493)Count pairs where arr[i] > 2×arr[j]
Count Smaller After Self (LC #315)For each element, count smaller elements to its right

All solved with the same modified merge sort template. Learn this one, solve them all.

📊 7. Complexity Analysis STEP 7 ▼
Time: O(n log n)
Space: O(n)

Same as Merge Sort — the inversion counting adds O(1) per merge comparison. No extra asymptotic cost.

☕ 8. Java Implementation STEP 8 ▼ 
public class CountInversions {

    public static long countInversions(int[] arr) {
        return mergeSort(arr, 0, arr.length - 1);
    }

    private static long mergeSort(int[] arr, int left, int right) {
        if (left >= right) return 0;

        int mid = left + (right - left) / 2;
        long count = 0;

        count += mergeSort(arr, left, mid);
        count += mergeSort(arr, mid + 1, right);
        count += merge(arr, left, mid, right);

        return count;
    }

    private static long merge(int[] arr, int left, int mid, int right) {
        int[] temp = new int[right - left + 1];
        int i = left, j = mid + 1, k = 0;
        long inversions = 0;

        while (i <= mid && j <= right) {
            if (arr[i] <= arr[j]) {
                temp[k++] = arr[i++];
            } else {
                inversions += (mid - i + 1); // THE KEY LINE
                temp[k++] = arr[j++];
            }
        }

        while (i <= mid) temp[k++] = arr[i++];
        while (j <= right) temp[k++] = arr[j++];

        System.arraycopy(temp, 0, arr, left, temp.length);
        return inversions;
    }

    public static void main(String[] args) {
        int[] arr = {2, 4, 1, 3, 5};
        System.out.println(countInversions(arr)); // 3
    }
}

💡 Literally one line added to Merge Sort (#24): inversions += (mid - i + 1). Everything else is identical.

⚠️ 10. Common Mistakes STEP 10 ▼
  1. Using int instead of long for count

    ❌ Max inversions = n(n-1)/2. For n=100,000 → ~5 billion → overflows int

    ✅ Always use long for inversion count

  2. Counting inversions without sorting

    ❌ Just counting without actually sorting → subsequent merge steps get wrong counts

    ✅ The array MUST be sorted during the process — counting depends on sorted halves

  3. Wrong inversion count formula

    ❌ inversions++ (counting one at a time) → misses bulk inversions

    ✅ inversions += (mid - i + 1) — ALL remaining left elements are inverted with current right element

  4. Forgetting to sum all three sources

    ✅ Total = left inversions + right inversions + split inversions (from merge)

🔑 11. When to Use This Pattern STEP 11 ▼

Signals to पहचान (identify):

  • ✅ "Count inversions" / "Count out-of-order pairs"
  • ✅ "How many swaps to sort?" (bubble sort swaps = inversions)
  • ✅ "Count pairs (i,j) where i < j and arr[i] > k × arr[j]" (Reverse Pairs)
  • ✅ "For each element, count smaller elements to its right"
  • ✅ Any cross-pair counting that can be decomposed into left/right/split
🚫 12. When NOT to Use STEP 12 ▼
  • ❌ "Count pairs with sum = target" — that's Two Sum (#6), not inversions
  • ❌ Non-comparison-based counting — modified merge sort only works for comparison-based pair counting
  • ❌ When you need the actual pairs, not just the count — merge sort gives count, not the pairs themselves
💡 "Modified Merge Sort = Merge Sort + one extra line. When right element is picked before left → inversions += remaining left elements. O(n log n) counting for free."
