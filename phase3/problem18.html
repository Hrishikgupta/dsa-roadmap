#18 — First Non-Repeating Char in Live Stream

🔖 Pattern: Queue + HashMap

🎯 Real-World Scenario START HERE ▼

Tum ek live chat app bana rahe ho. Characters ek-ek karke stream mein aa rahe hain:

Stream: a → a → b → c → b → d

Har character aane ke baad, batao: "Abhi tak ka first non-repeating character kaunsa hai?"

After receiving Stream so far First non-repeating Why
a"a"aOnly char, count=1
a"aa"nulla repeated, nothing unique
b"aab"ba repeated, b unique → b is first
c"aabc"bb came before c, both unique, b is first
b"aabcb"ca,b repeated → c is first unique
d"aabcbd"cc still first unique (came before d)

This is LeetCode #387 variant (stream version). Also appears as "First Unique Character in a String".

🧠 1. Problem Understanding STEP 1 ▼

Two versions of this problem:

Version Input Best Approach
Static (LC #387)Full string givenHashMap frequency count + scan
Stream (this problem)Chars arrive one by oneQueue + HashMap

Why Queue?

  • We need the first (earliest) non-repeating character
  • "First" = the one that arrived earliest = FIFO order
  • Queue maintains arrival order — front = earliest candidate
  • HashMap tracks frequency — tells us if a character is still unique

Stack vs Queue — when to use which:

  • Stack (LIFO): "most recent" — brackets (#15), next greater (#16), span (#17)
  • Queue (FIFO): "earliest/first" — this problem, BFS (later), task scheduling
💪 2. Brute Force Thinking STEP 2 ▼

Static version (full string):

// Simple: count frequencies, then scan for first with count=1 — O(n)
int[] count = new int[26];
for (char c : s.toCharArray()) count[c - 'a']++;
for (int i = 0; i < s.length(); i++) {
    if (count[s.charAt(i) - 'a'] == 1) return i;
}
return -1;

Two passes, O(n). Perfect for static input.

But for streaming:

  • After each new character, we'd need to re-scan the entire string to find the first unique → O(n) per query
  • For n characters total → O(n²)

🤔 Kya hum ek Queue maintain kar sakte hain of "candidates" (potentially unique chars), aur front se expired ones hata sakte hain?

⚡ 3. Optimization Thinking STEP 3 ▼

Key Insight — The Waiting Room Analogy:

Imagine a doctor's waiting room. Patients arrive and sit in order.

  • New patient arrives → sits at the back of the queue, marked as "first visit"
  • Same patient arrives again → marked as "repeat visitor" (in the frequency map)
  • Doctor calls: "Next unique patient please!" → check front of queue
  • If front patient is a repeat visitor → remove them, check next one
  • First non-repeat at the front = the answer

Two data structures working together:

  • Queue: maintains arrival order of characters (candidates for "first unique")
  • HashMap/int[26]: tracks frequency count of each character

On each new character:

  1. Increment its count in the frequency map
  2. Add it to the queue (if first time)
  3. While queue front has count > 1 → remove from front (it's no longer unique)
  4. Queue front = first non-repeating character (or queue empty = none)

💡 Pattern: Queue (FIFO order) + HashMap (frequency tracking) — lazy cleanup from front

🎯 4. Final Approach (Pattern-Based) STEP 4 ▼

Pattern: Queue + HashMap (Lazy Cleanup)

Algorithm for stream processing:

  1. Maintain Queue<Character> and int[26] count
  2. On each new character c:
    • count[c]++
    • queue.add(c)
    • While queue not empty AND count[queue.peek()] > 1: queue.poll()
  3. Answer = queue.isEmpty() ? null : queue.peek()

💡 "Lazy cleanup" = we don't remove chars from queue when they become repeated. We only clean up when they reach the front. This keeps the logic simple.

Notice: 'b' stays in the queue at position 2 even after becoming repeated. It only gets cleaned when it reaches the front (lazy cleanup).

⚙️ 6. Internal Mechanics STEP 6 ▼

Why "lazy cleanup" works:

  • We only care about the FRONT of the queue (the answer)
  • Repeated chars in the middle/back don't affect the answer
  • When they eventually reach the front, we clean them up then
  • Total cleanup across all operations = O(n) (each char added once, removed at most once)

What is a Queue in Java?

  • FIFO — First In, First Out
  • add(x) / offer(x) — add to back: O(1)
  • poll() — remove from front: O(1)
  • peek() — look at front without removing: O(1)
  • In Java: Queue<Character> q = new LinkedList<>()

📊 7. Complexity Analysis STEP 7 ▼

Stream version (per character):

Time (amortized): O(1)
Space: O(n)

Total for n characters: O(n) time, O(n) space.

Each character enters the queue once and leaves at most once → amortized O(1) per operation. Space bounded by unique characters (at most 26 for lowercase).

☕ 8. Java Implementation STEP 8 ▼

Stream version (Queue + HashMap):

import java.util.*;

public class FirstNonRepeatingStream {

    private Queue<Character> queue = new LinkedList<>();
    private int[] count = new int[26];

    public Character add(char c) {
        count[c - 'a']++;
        queue.add(c);

        // Lazy cleanup: remove repeated chars from front
        while (!queue.isEmpty() && count[queue.peek() - 'a'] > 1) {
            queue.poll();
        }

        return queue.isEmpty() ? null : queue.peek();
    }

    public static void main(String[] args) {
        FirstNonRepeatingStream stream = new FirstNonRepeatingStream();
        char[] input = {'a','a','b','c','b','d'};
        for (char c : input) {
            System.out.println("After '" + c + "': " + stream.add(c));
        }
        // a, null, b, b, c, c
    }
}

Static version (LC #387 — full string, find index):

public static int firstUniqChar(String s) {
    int[] count = new int[26];
    for (char c : s.toCharArray()) count[c - 'a']++;
    for (int i = 0; i < s.length(); i++) {
        if (count[s.charAt(i) - 'a'] == 1) return i;
    }
    return -1;
}

💡 Static version doesn't need a queue — two passes with int[26] is enough. Queue is only needed for streaming/online processing.

⚠️ 10. Common Mistakes STEP 10 ▼
  1. Removing from queue immediately when count becomes 2

    ❌ Searching and removing from middle of queue is O(n)

    ✅ Lazy cleanup — only remove from front. Middle elements get cleaned when they reach front.

  2. Not adding duplicate chars to queue

    It's fine to add them — they'll be cleaned up lazily. Alternatively, only add if count == 1 (first occurrence).

  3. Using Queue for the static version

    Overkill. Static version just needs two passes with int[26]. Queue is for streaming only.

🔑 11. When to Use This Pattern STEP 11 ▼

Signals to पहचान (identify):

  • ✅ "First non-repeating" / "First unique" in a stream
  • ✅ "Maintain order of arrival" + "filter by some condition"
  • ✅ Online/streaming data processing with FIFO requirement
  • ✅ "Sliding window" problems that need a deque (double-ended queue)

Queue appears again in:

  • BFS traversal (Problems #30-32) — the primary use of Queue in DSA
  • Task scheduling / Round Robin
  • Sliding window maximum (Monotonic Deque)
🚫 12. When NOT to Use STEP 12 ▼
  • ❌ Static string (full input given) — just use two-pass frequency count, no queue needed
  • ❌ "Most recent unique" — that's LIFO (Stack), not FIFO (Queue)
  • ❌ "K-th non-repeating" — queue gives first, not k-th (need modification)
💡 "First unique in a stream = Queue (FIFO order) + HashMap (frequency). Lazy cleanup from front — don't search the middle."
