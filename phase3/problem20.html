#20 — Power of Recursion — Tower of Hanoi

🔖 Pattern: Recursion (Trust the Function)

🎯 Real-World Scenario START HERE ▼

Ek ancient temple mein 3 pillars hain (A, B, C). Pillar A pe 3 golden disks hain, smallest on top:

  Pillar A      Pillar B      Pillar C
    |              |              |
   ===            |              |
  =====           |              |
 =======          |              |
___________  ___________  ___________

Goal: Move all 3 disks from A to C.

Rules:

  1. Only one disk can be moved at a time
  2. Only the top disk of a pillar can be moved
  3. A larger disk can NEVER be placed on a smaller disk

This puzzle is impossible to solve iteratively by thinking step-by-step. But with recursion, it becomes 3 lines of logic.

🧠 1. Problem Understanding STEP 1 ▼

What is Recursion?

  • A function that calls itself to solve a smaller version of the same problem
  • Every recursive solution has two parts:
    • Base case: the simplest version that can be solved directly (stops recursion)
    • Recursive case: break the problem into smaller subproblems + call yourself

Why is this a paradigm shift?

Phase 1-2 (Iterative) Phase 3+ (Recursive)
You control every stepYou define the pattern, recursion handles the steps
Loops (for, while)Function calls itself
Think: "how to do each step"Think: "if smaller problem is solved, how do I use it?"

The #1 rule of recursion:

🧘 "Trust the function." — Assume the recursive call works correctly. Don't trace every call in your head. Just define: base case + how to use the smaller solution.
💪 2. Brute Force Thinking STEP 2 ▼

Trying to think iteratively:

"Move disk 1 to C, then disk 2 to B, then disk 1 to B, then disk 3 to C, then..."

Your brain explodes at n=4. At n=10, it's impossible to track manually.

Recursive thinking — the breakthrough:

Don't think about individual moves. Think about the STRUCTURE:

  • To move n disks from A to C using B as helper:
  • Step 1: Move top n-1 disks from A to B (using C as helper) — trust this works!
  • Step 2: Move the bottom (largest) disk from A to C — one simple move
  • Step 3: Move n-1 disks from B to C (using A as helper) — trust this works!

🤔 "But how do steps 1 and 3 work?" — That's the beauty. They work the SAME WAY, just with smaller n. Eventually n=1 (base case) and you just move one disk directly.

⚡ 3. Optimization Thinking STEP 3 ▼

Key Insight — The Manager Analogy:

You're a manager. You need to move a stack of 100 files from desk A to desk C.

  • You tell your assistant: "Move the top 99 files to desk B." (You don't care HOW — that's their problem)
  • You move the bottom file from A to C yourself. (One simple action)
  • You tell your assistant: "Now move those 99 files from B to C."

Your assistant does the same thing — tells THEIR assistant to handle 98 files. And so on, until someone has just 1 file (base case — just move it).

The recursive formula:

hanoi(n, from, to, helper):
    if n == 1:                          // Base case
        move disk from → to
        return
    hanoi(n-1, from, helper, to)        // Move n-1 to helper
    move disk from → to                  // Move largest disk
    hanoi(n-1, helper, to, from)        // Move n-1 from helper to target

💡 3 lines of logic. Solves any n. That's the power of recursion.

🎯 4. Final Approach (Pattern-Based) STEP 4 ▼

Pattern: Recursion — "Trust the Function"

The recursive thinking template:

  1. Define the function signature: What does it take? What does it do?
    • hanoi(n, from, to, helper) — moves n disks from → to using helper
  2. Base case: What's the simplest version?
    • n == 1 → just move the disk directly
  3. Recursive case: Assume the function works for n-1. How do you solve for n?
    • Move n-1 disks out of the way (to helper)
    • Move the big disk (to target)
    • Move n-1 disks back (from helper to target)
  4. Trust: Don't trace the recursion. Trust that step 1 and 3 work.

7 moves for 3 disks. Formula: 2ⁿ - 1 moves. For n=3: 2³-1 = 7 ✓

Move 4 (the middle) is the key moment — the largest disk finally moves. Everything before prepares for it, everything after cleans up.

⚙️ 6. Internal Mechanics STEP 6 ▼

How recursion works under the hood — the Call Stack:

  • Every function call creates a "frame" on the call stack (like Stack data structure from #15!)
  • Each frame stores: function parameters, local variables, return address
  • When function returns → frame is popped, execution resumes in the caller
// Call stack for hanoi(3, A, C, B):
// At the deepest point (move 1):

| hanoi(1, A, C, B)  |  ← top (executing)
| hanoi(2, A, B, C)  |  ← waiting for sub-call
| hanoi(3, A, C, B)  |  ← waiting for sub-call
|____________________|  ← bottom (main)

Max stack depth = n. For n=3, stack goes 3 levels deep.

The three pillars of recursive thinking:

  1. Base case — when to stop (n == 1)
  2. Recursive case — how to break down (n → n-1)
  3. Trust — believe the function works for smaller inputs. Don't trace every call.

📊 7. Complexity Analysis STEP 7 ▼
Time: O(2ⁿ)
Space: O(n)

Time: T(n) = 2·T(n-1) + 1 → T(n) = 2ⁿ - 1 moves. Exponential — but this is provably optimal for Tower of Hanoi.

Space: O(n) — recursion depth = n (call stack frames).

Not all recursive solutions are exponential. Merge Sort (#24) is O(n log n). Recursion is a technique, not a complexity class.

☕ 8. Java Implementation STEP 8 ▼ 
public class TowerOfHanoi {

    public static void hanoi(int n, char from, char to, char helper) {
        if (n == 1) {
            System.out.println("Move disk 1 from " + from + " to " + to);
            return;
        }

        hanoi(n - 1, from, helper, to);     // Move n-1 to helper
        System.out.println("Move disk " + n + " from " + from + " to " + to);
        hanoi(n - 1, helper, to, from);     // Move n-1 from helper to target
    }

    public static void main(String[] args) {
        hanoi(3, 'A', 'C', 'B');
    }
}

Output:

Move disk 1 from A to C
Move disk 2 from A to B
Move disk 1 from C to B
Move disk 3 from A to C
Move disk 1 from B to A
Move disk 2 from B to C
Move disk 1 from A to C

💡 The entire solution is 6 lines of actual logic. Recursion handles all the complexity.

⚠️ 10. Common Mistakes STEP 10 ▼
  1. Trying to trace every recursive call

    ❌ "Let me follow hanoi(3)→hanoi(2)→hanoi(1)→..." — brain overload

    ✅ Trust the function. Just verify: base case correct? Recursive case uses smaller input? Combines correctly?

  2. Missing the base case

    ❌ No if (n == 1) → infinite recursion → StackOverflowError

    ✅ Always define the base case FIRST

  3. Wrong parameter order in recursive calls

    ❌ Swapping 'to' and 'helper' incorrectly → disks end up on wrong pillar

    ✅ Think: "where am I moving FROM, TO, and what's the HELPER?" for each call

  4. Confusing recursion with iteration

    Recursion isn't "a loop that calls itself." It's "solving a problem by solving a smaller version of itself."

🔑 11. When to Use This Pattern STEP 11 ▼

Signals to पहचान (identify):

  • ✅ Problem has a natural "smaller version of itself" structure
  • ✅ Tree/graph traversal (every subtree is a smaller tree)
  • ✅ "Generate all combinations/permutations/subsets"
  • ✅ "Divide and conquer" — split, solve halves, merge
  • ✅ Problem says "recursive" or has nested/hierarchical structure

Recursion is the FOUNDATION for:

  • Subsets, Permutations, Combinations (#21-23)
  • Merge Sort, Quick Sort (#24-25)
  • Tree DFS (#26-30)
  • Graph DFS (#31-34)
  • Backtracking (#36-38)
  • Dynamic Programming (#39-50)
🚫 12. When NOT to Use STEP 12 ▼
  • ❌ Simple iteration works — don't use recursion for summing an array or finding max
  • ❌ Very deep recursion (n > 10,000) — risk of StackOverflowError. Convert to iterative with explicit stack.
  • ❌ Overlapping subproblems without memoization — leads to exponential time (that's where DP comes in, Phase 5)
💡 "Recursion = Trust the function. Define base case. Define how to use the smaller solution. Don't trace — trust."

Next problem mein recursion ka pehla practical pattern aata hai: Include/Exclude.

"Generate all subsets of a menu" — for each item, you have two choices: include it or skip it.

  • Menu: [A, B, C]
  • Subsets: [], [A], [B], [C], [AB], [AC], [BC], [ABC] — total 2ⁿ = 8
  • At each item: branch into "take" and "skip" → binary tree of decisions

This is the foundation for backtracking (Phase 4) and DP (Phase 5).
