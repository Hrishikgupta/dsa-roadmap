#19 — LRU Cache for Browser History

🔖 Pattern: HashMap + Doubly Linked List

🎯 Real-World Scenario START HERE ▼

Tum ek browser bana rahe ho. Browser cache mein sirf 3 pages fit hote hain.

User visits pages in this order:

  1. Visit Google → Cache: [Google]
  2. Visit YouTube → Cache: [Google, YouTube]
  3. Visit GitHub → Cache: [Google, YouTube, GitHub] (full!)
  4. Visit Twitter → Cache full! Evict Google (least recently used) → [YouTube, GitHub, Twitter]
  5. Visit YouTube again → Move to most recent → [GitHub, Twitter, YouTube]
  6. Visit Reddit → Evict GitHub (least recently used) → [Twitter, YouTube, Reddit]

Two operations needed, both in O(1):

  • get(key) — fetch page from cache (and mark as recently used)
  • put(key, value) — add/update page (evict LRU if full)

This is LeetCode #146 — LRU Cache. Top interview question at Google, Amazon, Meta, Microsoft. Also a core system design concept.

🧠 1. Problem Understanding STEP 1 ▼

What exactly is being asked?

  • Design a data structure with fixed capacity
  • get(key): return value if key exists (mark as recently used), else return -1
  • put(key, value): insert or update. If at capacity, evict the least recently used item first.
  • Both operations must be O(1)

Why is O(1) hard?

Data Structure get put evict LRU move to recent
HashMap aloneO(1)O(1)O(n) — no order infoO(n)
Linked List aloneO(n) — searchO(1) — add to endO(1) — remove headO(n) — find node
HashMap + DLLO(1)O(1)O(1)O(1)

💡 Neither alone gives O(1) for everything. Together they complement each other perfectly.

💪 2. Brute Force Thinking STEP 2 ▼

Approach 1: ArrayList + timestamp

  • Store (key, value, lastUsedTime) in a list
  • On eviction: scan for minimum timestamp → O(n)

Approach 2: LinkedHashMap (Java cheat)

// Java's LinkedHashMap with access-order = true does LRU automatically
class LRUCache extends LinkedHashMap<Integer, Integer> {
    int capacity;
    LRUCache(int cap) {
        super(cap, 0.75f, true); // true = access order
        this.capacity = cap;
    }
    protected boolean removeEldestEntry(Map.Entry e) {
        return size() > capacity;
    }
}

⚠️ Interviewers want you to build it from scratch, not use LinkedHashMap. They want to see you design the data structure.

⚡ 3. Optimization Thinking STEP 3 ▼

Key Insight — Two data structures, each covering the other's weakness:

  • HashMap: O(1) lookup by key → solves "find the node quickly"
  • Doubly Linked List: O(1) insert/delete at any position (given the node) → solves "reorder quickly"
  • HashMap stores key → Node (pointer to the DLL node)
  • DLL maintains usage order: head = LRU (least recent), tail = MRU (most recent)

Why Doubly Linked List, not Singly?

  • To remove a node from the middle, you need access to its previous node
  • Singly linked: need to traverse from head to find previous → O(n)
  • Doubly linked: node.prev gives previous directly → O(1)

The design:

HashMap: key → Node
DLL:    HEAD ↔ node1 ↔ node2 ↔ node3 ↔ TAIL
        (LRU)                          (MRU)
        dummy                          dummy

💡 Dummy head and tail nodes simplify edge cases — no null checks for empty list or single element.

💡 Pattern: HashMap (O(1) lookup) + Doubly Linked List (O(1) reorder) = O(1) LRU Cache

🎯 4. Final Approach (Pattern-Based) STEP 4 ▼

Operations breakdown:

get(key):

  1. If key not in map → return -1
  2. Get node from map
  3. Move node to tail (most recently used)
  4. Return node.value

put(key, value):

  1. If key exists → update value, move to tail
  2. If key doesn't exist:
    • If at capacity → remove head.next (LRU), remove from map
    • Create new node, add to tail, add to map

Helper operations on DLL:

  • addToTail(node): insert before dummy tail
  • removeNode(node): unlink from current position
  • moveToTail(node): removeNode + addToTail

⚙️ 6. Internal Mechanics STEP 6 ▼

DLL node removal in O(1) — the magic:

// Remove node from its current position
node.prev.next = node.next;
node.next.prev = node.prev;
// That's it. Two pointer updates. O(1).

This is impossible with a singly linked list (you'd need to find node.prev by traversing from head).

Why dummy head and tail?

// Without dummies: need null checks everywhere
if (node.prev != null) node.prev.next = node.next;
else head = node.next; // was the head
if (node.next != null) node.next.prev = node.prev;
else tail = node.prev; // was the tail

// With dummies: always works, no special cases
node.prev.next = node.next;
node.next.prev = node.prev;

Dummy nodes are a classic technique — they eliminate edge cases for empty list, single element, head/tail operations.

Real-world LRU caches:

  • CPU cache (L1, L2, L3) — hardware LRU
  • Redis — in-memory cache with LRU eviction policy
  • Browser cache — exactly this problem
  • CDN edge caches — cache popular content, evict stale
  • Database buffer pool — cache frequently accessed pages
📊 7. Complexity Analysis STEP 7 ▼
get(): O(1)
put(): O(1)
Space: O(capacity)

HashMap lookup O(1) + DLL insert/remove O(1) = O(1) per operation. Space = capacity nodes + capacity map entries.

☕ 8. Java Implementation STEP 8 ▼ 
import java.util.HashMap;

public class LRUCache {

    class Node {
        int key, val;
        Node prev, next;
        Node(int k, int v) { key = k; val = v; }
    }

    private int capacity;
    private HashMap<Integer, Node> map;
    private Node head, tail; // dummy nodes

    public LRUCache(int capacity) {
        this.capacity = capacity;
        map = new HashMap<>();
        head = new Node(0, 0);
        tail = new Node(0, 0);
        head.next = tail;
        tail.prev = head;
    }

    public int get(int key) {
        if (!map.containsKey(key)) return -1;
        Node node = map.get(key);
        moveToTail(node);
        return node.val;
    }

    public void put(int key, int value) {
        if (map.containsKey(key)) {
            Node node = map.get(key);
            node.val = value;
            moveToTail(node);
        } else {
            if (map.size() == capacity) {
                Node lru = head.next;
                removeNode(lru);
                map.remove(lru.key);
            }
            Node node = new Node(key, value);
            addToTail(node);
            map.put(key, node);
        }
    }

    private void addToTail(Node node) {
        node.prev = tail.prev;
        node.next = tail;
        tail.prev.next = node;
        tail.prev = node;
    }

    private void removeNode(Node node) {
        node.prev.next = node.next;
        node.next.prev = node.prev;
    }

    private void moveToTail(Node node) {
        removeNode(node);
        addToTail(node);
    }
}

💡 The Node stores the key too — needed when evicting LRU to remove it from the HashMap. Without the key in the node, you can't do map.remove(lru.key).

⚠️ 10. Common Mistakes STEP 10 ▼
  1. Not storing key in the Node

    ❌ When evicting LRU, you need map.remove(lru.key) — without key in node, you can't do this in O(1)

  2. Forgetting to move to tail on get()

    ❌ get() should mark as recently used — if you don't move, eviction order is wrong

  3. Not handling put() for existing key

    ❌ Adding a duplicate node → map points to new node, old node orphaned in DLL

    ✅ Check if key exists first → update value + move to tail

  4. Using singly linked list

    ❌ Can't remove a node in O(1) without access to previous

    ✅ Must be doubly linked for O(1) removal

🔑 11. When to Use This Pattern STEP 11 ▼

Signals to पहचान (identify):

  • ✅ "Design a cache with eviction policy"
  • ✅ "O(1) get and put with capacity limit"
  • ✅ "Least recently used" / "Most recently used"
  • ✅ Any problem needing O(1) lookup + O(1) ordered insertion/deletion

Variants:

  • LFU Cache (Least Frequently Used) — LC #460 — harder, needs frequency buckets
  • TTL Cache — evict based on time, not usage
  • LRU with expiry — combine LRU + timestamp
🚫 12. When NOT to Use STEP 12 ▼
  • ❌ No capacity limit — just use a HashMap
  • ❌ Eviction based on frequency, not recency — use LFU instead
  • ❌ Need sorted order — use TreeMap
💡 "LRU Cache = HashMap (O(1) find) + Doubly Linked List (O(1) reorder). Node stores key too — needed for eviction cleanup. Dummy head/tail eliminate edge cases."
