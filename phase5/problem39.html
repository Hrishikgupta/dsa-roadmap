#39 — Climbing Stairs — How Many Ways?

🔖 Pattern: DP (Fibonacci Pattern)

🎯 Real-World Scenario START HERE ▼

Tum ek staircase pe ho with n steps. Har baar tum 1 step ya 2 steps chadh sakte ho.

"Top tak pahunchne ke kitne distinct ways hain?"

n=1: [1]                           → 1 way
n=2: [1,1] [2]                     → 2 ways
n=3: [1,1,1] [1,2] [2,1]          → 3 ways
n=4: [1,1,1,1] [1,1,2] [1,2,1] [2,1,1] [2,2] → 5 ways

1, 2, 3, 5, 8, 13... — it's Fibonacci!

This is LeetCode #70 — Climbing Stairs. The gentlest intro to Dynamic Programming.

🧠 1. Problem Understanding STEP 1 ▼

What is Dynamic Programming (DP)?

  • DP = Recursion + Memoization (remember what you've already computed)
  • Or equivalently: build the answer bottom-up from smaller subproblems
  • Works when a problem has overlapping subproblems (same subproblem solved multiple times)

Why this problem is DP:

  • To reach step n, you came from step n-1 (took 1 step) or step n-2 (took 2 steps)
  • So: ways(n) = ways(n-1) + ways(n-2)
  • This is the Fibonacci recurrence! And it has overlapping subproblems:
ways(5) = ways(4) + ways(3)
ways(4) = ways(3) + ways(2)  ← ways(3) computed TWICE!
ways(3) = ways(2) + ways(1)  ← ways(2) computed TWICE!

Three approaches (evolution of thinking):

Approach Time Space Idea
Plain RecursionO(2ⁿ)O(n)Recomputes same subproblems
Top-down DP (Memoization)O(n)O(n)Recursion + cache results
Bottom-up DP (Tabulation)O(n)O(1)*Build from base cases up

*O(1) because we only need the last 2 values, not the entire array.

💪 2. Brute Force Thinking STEP 2 ▼

Plain recursion — O(2ⁿ):

int climbStairs(int n) {
    if (n <= 2) return n;
    return climbStairs(n - 1) + climbStairs(n - 2);
}

Works but exponential — for n=45, takes billions of calls. The problem: climbStairs(3) is computed millions of times.

🤔 What if we REMEMBER the result of climbStairs(3) the first time, and just look it up next time? That's memoization — and that's DP.

⚡ 3. Optimization Thinking STEP 3 ▼

Step 1: Top-down DP (Memoization) — add a cache to recursion:

int[] memo = new int[n + 1];
int climb(int n) {
    if (n <= 2) return n;
    if (memo[n] != 0) return memo[n]; // already computed!
    memo[n] = climb(n-1) + climb(n-2);
    return memo[n];
}

O(n) time, O(n) space. Each subproblem computed exactly once.

Step 2: Bottom-up DP (Tabulation) — no recursion, just a loop:

int[] dp = new int[n + 1];
dp[1] = 1; dp[2] = 2;
for (int i = 3; i <= n; i++)
    dp[i] = dp[i-1] + dp[i-2];
return dp[n];

Step 3: Space-optimized — only need last 2 values:

int a = 1, b = 2;
for (int i = 3; i <= n; i++) {
    int c = a + b;
    a = b; b = c;
}
return b;

O(n) time, O(1) space. Optimal.

💡 DP thinking: Define recurrence → Memoize (top-down) → Tabulate (bottom-up) → Optimize space.

🎯 4. Final Approach STEP 4 ▼

The DP recipe (use for EVERY DP problem):

  1. Define state: What does dp[i] represent? → "number of ways to reach step i"
  2. Recurrence: How does dp[i] relate to smaller subproblems? → dp[i] = dp[i-1] + dp[i-2]
  3. Base cases: What are the smallest subproblems? → dp[1] = 1, dp[2] = 2
  4. Order: Fill dp from small to large (bottom-up) or recurse with memo (top-down)
  5. Answer: Which dp value is the answer? → dp[n]

This 5-step recipe works for ALL DP problems in Phase 5. The hard part is step 1 and 2 — defining state and recurrence.

☕ 8. Java Implementation STEP 8 ▼

Bottom-up (space-optimized):

public class ClimbingStairs {

    public static int climbStairs(int n) {
        if (n <= 2) return n;
        int a = 1, b = 2;
        for (int i = 3; i <= n; i++) {
            int c = a + b;
            a = b;
            b = c;
        }
        return b;
    }
}

💡 5 lines of logic. From O(2ⁿ) recursion to O(n)/O(1) DP — that's the power of dynamic programming.

⚠️ 10. Common Mistakes STEP 10 ▼
  1. Using plain recursion without memoization

    ❌ O(2ⁿ) — TLE for n > 30

    ✅ Always memoize or use bottom-up

  2. Wrong base cases

    ❌ dp[0] = 0, dp[1] = 1 → dp[2] = 1 (wrong, should be 2)

    ✅ dp[1] = 1, dp[2] = 2 (or dp[0] = 1, dp[1] = 1 depending on convention)

  3. Not recognizing the Fibonacci pattern

    Many DP problems are "Fibonacci in disguise" — decode message, tiling, etc. Recognize the recurrence.

🔑 11. When to Use This Pattern STEP 11 ▼
  • ✅ "How many ways to reach ___?"
  • ✅ "Minimum/maximum cost to reach ___?"
  • ✅ Problem has overlapping subproblems (same computation repeated)
  • ✅ Problem has optimal substructure (optimal solution uses optimal sub-solutions)
  • ✅ Recurrence relation can be defined: dp[i] = f(dp[i-1], dp[i-2], ...)
🚫 12. When NOT to Use STEP 12 ▼
  • ❌ No overlapping subproblems — plain recursion or greedy is enough
  • ❌ "Generate all solutions" — DP counts/optimizes, backtracking generates
  • ❌ Problem doesn't have optimal substructure — greedy might work instead
💡 "DP = recursion + memory. Define state → write recurrence → base cases → fill bottom-up → optimize space. This recipe solves every DP problem."
