#12 — Smallest Window Containing All Keywords

🔖 Pattern: Sliding Window (Min Window Substring)

🎯 Real-World Scenario START HERE ▼

Tum ek search engine bana rahe ho. User ne search kiya: "ABC"

Document text: "ADOBECODEBANC"

Tumhe document mein sabse chhota snippet dikhana hai jismein search ke saare keywords (A, B, C) present hon.

Possible windows containing A, B, C:

  • "ADOBEC" (index 0-5, length 6) ✓
  • "ADOBECODEBANC" — wait, "BANC" (index 9-12, length 4) ✓

Answer: "BANC" — smallest window containing all of A, B, C.

This is LeetCode #76 — Minimum Window Substring. Considered one of the hardest sliding window problems. Top FAANG interview question.

🧠 1. Problem Understanding STEP 1 ▼

What exactly is being asked?

  • Given string s and string t, find the minimum-length substring of s that contains ALL characters of t (including duplicates)
  • If t = "AAB", the window must contain at least 2 A's and 1 B
  • If no such window exists, return ""

Input → Output:

  • Input: String s, String t
  • Output: String — minimum window substring (or "")

This combines THREE patterns we've learned:

💪 2. Brute Force Thinking STEP 2 ▼

First instinct: Check every substring of s, see if it contains all chars of t.

// Brute Force — O(n² × m)
for each start i:
  for each end j >= i:
    if substring(i,j) contains all chars of t:
      update minWindow

O(n²) substrings × O(m) containment check = O(n² × m). Way too slow.

🤔 Problem #9 mein humne seekha: variable window expand karo jab condition not met, shrink karo jab met. Same idea — but condition is "all chars of t are present in window".

⚡ 3. Optimization Thinking STEP 3 ▼

Key Insight — The "Shopping List" Analogy:

Imagine you're in a supermarket aisle (string s). You have a shopping list (string t): need 1 Apple, 1 Banana, 1 Cherry.

  • Walk forward (expand right), picking up items as you see them
  • Once your cart has everything on the list → you've found a valid window
  • Now try to drop items from the back of your path (shrink left) to find the shortest valid stretch
  • Keep walking and shrinking to find the overall shortest

The clever trick — a "formed" counter:

  • Instead of comparing two full frequency maps every time (expensive), track a single number: formed
  • formed = how many UNIQUE characters in t have their required count met in the window
  • When formed == required (total unique chars in t) → window is valid
  • This turns the "is window valid?" check from O(m) to O(1)

💡 Pattern: Variable Sliding Window + Two Frequency Maps + Formed Counter

🎯 4. Final Approach (Pattern-Based) STEP 4 ▼

Data structures needed:

  • tFreq — frequency map of t (what we need)
  • windowFreq — frequency map of current window (what we have)
  • required — number of unique chars in t
  • formed — how many unique chars currently meet their required count

Algorithm:

  1. Build tFreq from t. Set required = tFreq.size()
  2. left = 0, formed = 0, track best window (start, length)
  3. For right = 0 to s.length() - 1:
    • Add s.charAt(right) to windowFreq
    • If this char is in tFreq AND windowFreq[char] == tFreq[char] → formed++
    • While formed == required (window is valid):
      • Update best window if current is smaller
      • Remove s.charAt(left) from windowFreq
      • If this char is in tFreq AND windowFreq[char] < tFreq[char] → formed--
      • left++
  4. Return best window substring (or "" if never found)

⚙️ 6. Internal Mechanics STEP 6 ▼

Why the "formed" counter is genius:

  • Without it: every time we add/remove a char, we'd compare all 26+ entries in both maps → O(m) per step
  • With it: we only update formed when a specific char's count crosses the threshold → O(1) per step
  • formed++ when windowFreq[c] == tFreq[c] (just met the requirement)
  • formed-- when windowFreq[c] < tFreq[c] (just lost the requirement)

Handling duplicates in t:

  • If t = "AABC", then tFreq = {A:2, B:1, C:1}, required = 3 (3 unique chars)
  • Window needs at least 2 A's for the A requirement to be "formed"
  • The frequency comparison handles this naturally — windowFreq[A] == tFreq[A] means 2 == 2

📊 7. Complexity Analysis STEP 7 ▼
Time: O(|s| + |t|)
Space: O(|s| + |t|)

Breakdown:

  • Building tFreq: O(|t|)
  • Sliding window: each char in s is added once and removed at most once → O(2|s|) = O(|s|)
  • Space: two HashMaps, each at most O(|s| + |t|) entries

In practice, if charset is fixed (e.g., 128 ASCII), space is O(1). The formed counter keeps the per-step work O(1).

☕ 8. Java Implementation STEP 8 ▼ 
import java.util.HashMap;

public class MinWindowSubstring {

    public static String minWindow(String s, String t) {
        if (s.length() < t.length()) return "";

        // Build frequency map for t
        HashMap<Character, Integer> tFreq = new HashMap<>();
        for (char c : t.toCharArray()) {
            tFreq.put(c, tFreq.getOrDefault(c, 0) + 1);
        }

        int required = tFreq.size();
        int formed = 0;
        HashMap<Character, Integer> windowFreq = new HashMap<>();

        int left = 0;
        int minLen = Integer.MAX_VALUE;
        int minStart = 0;

        for (int right = 0; right < s.length(); right++) {
            char c = s.charAt(right);
            windowFreq.put(c, windowFreq.getOrDefault(c, 0) + 1);

            // Check if this char's requirement is now met
            if (tFreq.containsKey(c) &&
                windowFreq.get(c).intValue() == tFreq.get(c).intValue()) {
                formed++;
            }

            // Shrink window while valid
            while (formed == required) {
                if (right - left + 1 < minLen) {
                    minLen = right - left + 1;
                    minStart = left;
                }

                char leftChar = s.charAt(left);
                windowFreq.put(leftChar, windowFreq.get(leftChar) - 1);

                if (tFreq.containsKey(leftChar) &&
                    windowFreq.get(leftChar) < tFreq.get(leftChar)) {
                    formed--;
                }
                left++;
            }
        }

        return minLen == Integer.MAX_VALUE
            ? ""
            : s.substring(minStart, minStart + minLen);
    }

    public static void main(String[] args) {
        System.out.println(minWindow("ADOBECODEBANC", "ABC")); // "BANC"
        System.out.println(minWindow("a", "a"));               // "a"
        System.out.println(minWindow("a", "aa"));              // ""
    }
}

⚠️ Critical: Use .intValue() for Integer comparison, not ==. Java caches Integer objects only for -128 to 127. Beyond that, == compares references, not values.

⚠️ 10. Common Mistakes STEP 10 ▼
  1. Using == to compare Integer objects

    ❌ windowFreq.get(c) == tFreq.get(c) — fails for values > 127

    ✅ windowFreq.get(c).intValue() == tFreq.get(c).intValue() or .equals()

  2. Incrementing formed for every matching char (not just when threshold is crossed)

    ❌ if (tFreq.containsKey(c)) formed++ — formed goes way above required

    ✅ Only increment when windowFreq[c] == tFreq[c] (exact threshold crossing)

  3. Not handling duplicates in t

    ❌ Using HashSet for t — loses frequency info. t="AAB" needs 2 A's.

    ✅ Always use frequency map for t

  4. Forgetting to track minStart

    ❌ Only tracking minLen — can't reconstruct the actual substring

    ✅ Track both minStart and minLen

🔑 11. When to Use This Pattern STEP 11 ▼

Signals to पहचान (identify) this pattern:

  • ✅ "Minimum window containing all characters of T"
  • ✅ "Smallest substring with all required elements"
  • ✅ "Find all anagram substrings" (LC #438 — fixed window variant)
  • ✅ "Permutation in string" (LC #567 — is any permutation of t a substring of s?)
  • ✅ Any "find substring containing all of X" problem
🚫 12. When NOT to Use STEP 12 ▼
  • ❌ Order of characters matters — "ABC" must appear in that order → use two pointers or DP, not this
  • ❌ Subsequence, not substring — non-contiguous matching needs different approach
  • ❌ Pattern matching (regex-like) — different problem entirely
