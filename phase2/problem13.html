#13 — Trapping Rainwater Between Buildings

🔖 Pattern: Two Pointer (Opposite Ends) + Pre-computation

🎯 Real-World Scenario START HERE ▼

Imagine a city skyline after heavy rain. Buildings of different heights are lined up:

[0, 1, 0, 2, 1, 0, 1, 3, 2, 1, 2, 1]

Rain falls and water gets trapped between taller buildings:

   █   ██ █

Water fills the gaps:
       █
   █ ~ ██ █
 █ ██ ████ █
 ~ = water trapped

Question: Total kitna water trap hoga?

Answer: 6 units

This is LeetCode #42 — Trapping Rain Water. One of the most classic hard problems. Asked at Google, Amazon, Microsoft constantly.

🧠 1. Problem Understanding STEP 1 ▼

What exactly is being asked?

  • Given an array where each element represents the height of a bar/building
  • Calculate how much water can be trapped between the bars after rain

Input → Output:

  • Input: int[] height
  • Output: int — total units of trapped water

The core insight — water at any position:

Water trapped at index i = min(maxLeft, maxRight) - height[i]

  • maxLeft = tallest building to the LEFT of i (including i)
  • maxRight = tallest building to the RIGHT of i (including i)
  • Water level at i = the shorter of the two walls
  • Water trapped at i = water level - building height (if positive)

Connection to previous problems:

  • #3 (Prefix Sum): pre-computing cumulative data from left → same idea but with max instead of sum
  • #6 (Two Pointer pair sum): opposite-end pointers → same structure but different logic
💪 2. Brute Force Thinking STEP 2 ▼

First instinct: For each position, scan left for max and scan right for max.

// Brute Force — O(n²)
int total = 0;
for (int i = 0; i < n; i++) {
    int maxLeft = 0, maxRight = 0;
    for (int j = 0; j <= i; j++) maxLeft = Math.max(maxLeft, height[j]);
    for (int j = i; j < n; j++) maxRight = Math.max(maxRight, height[j]);
    total += Math.min(maxLeft, maxRight) - height[i];
}

O(n) per position × n positions = O(n²).

🤔 Hum har position ke liye left-max aur right-max baar baar calculate kar rahe hain. Kya pehle se pre-compute kar sakte hain? (Prefix Sum #3 jaisa thinking!)

⚡ 3. Optimization Thinking STEP 3 ▼

Approach A: Pre-compute arrays — O(n) time, O(n) space

  • Build leftMax[i] = max height from 0 to i (scan left → right)
  • Build rightMax[i] = max height from i to n-1 (scan right → left)
  • Water at i = min(leftMax[i], rightMax[i]) - height[i]

This is the "prefix max" approach — same idea as prefix sum but tracking max instead of sum.

Approach B: Two Pointer — O(n) time, O(1) space ⭐

Can we avoid the two extra arrays?

  • Use two pointers: left = 0, right = n-1
  • Track leftMax and rightMax as we go
  • Key insight: we only need the smaller of the two maxes
  • If leftMax < rightMax: water at left is determined by leftMax (the bottleneck). Process left, move left++.
  • If rightMax <= leftMax: water at right is determined by rightMax. Process right, move right--.

💡 Pattern: Two Pointer from opposite ends + running max tracking — O(n) time, O(1) space

🎯 4. Final Approach (Pattern-Based) STEP 4 ▼

Pattern: Two Pointer (Opposite Ends) with Running Max

Why it works:

  • If height[left] < height[right]: the right side is guaranteed to have a wall ≥ height[right]
  • So the bottleneck for position left is leftMax — we know it completely
  • We don't need to know the exact rightMax — we just know it's big enough
  • Same logic flipped for the right side

Algorithm:

  1. left = 0, right = n-1, leftMax = 0, rightMax = 0, water = 0
  2. While left < right:
    • If height[left] < height[right]:
      • If height[left] >= leftMax: update leftMax
      • Else: water += leftMax - height[left]
      • left++
    • Else:
      • If height[right] >= rightMax: update rightMax
      • Else: water += rightMax - height[right]
      • right--

⚙️ 6. Internal Mechanics STEP 6 ▼

Why processing the shorter side is safe:

  • If height[left] < height[right], we know there's a wall on the right that's at least height[right] tall
  • So the water at left is bounded by leftMax (we know it exactly) and something ≥ height[right] on the right
  • Since height[left] < height[right], the bottleneck is definitely leftMax
  • We don't need to know the exact rightMax — just that it's big enough

Three approaches compared:

Time: O(n)
Space: O(1)

Single pass. Two pointers meet in the middle. Only 4 variables tracked (left, right, leftMax, rightMax).

☕ 8. Java Implementation STEP 8 ▼

Two Pointer — O(n) time, O(1) space:

public class TrappingRainWater {

    public static int trap(int[] height) {
        int left = 0, right = height.length - 1;
        int leftMax = 0, rightMax = 0;
        int water = 0;

        while (left < right) {
            if (height[left] < height[right]) {
                if (height[left] >= leftMax) {
                    leftMax = height[left];
                } else {
                    water += leftMax - height[left];
                }
                left++;
            } else {
                if (height[right] >= rightMax) {
                    rightMax = height[right];
                } else {
                    water += rightMax - height[right];
                }
                right--;
            }
        }
        return water;
    }

    public static void main(String[] args) {
        int[] h = {0,1,0,2,1,0,1,3,2,1,2,1};
        System.out.println(trap(h)); // 6
    }
}

Prefix Max approach (easier to understand, O(n) space):

public static int trapPrefixMax(int[] height) {
    int n = height.length;
    int[] leftMax = new int[n];
    int[] rightMax = new int[n];

    leftMax[0] = height[0];
    for (int i = 1; i < n; i++)
        leftMax[i] = Math.max(leftMax[i-1], height[i]);

    rightMax[n-1] = height[n-1];
    for (int i = n-2; i >= 0; i--)
        rightMax[i] = Math.max(rightMax[i+1], height[i]);

    int water = 0;
    for (int i = 0; i < n; i++)
        water += Math.min(leftMax[i], rightMax[i]) - height[i];

    return water;
}

💡 Interview tip: Explain the prefix max approach first (easier to reason about), then optimize to two pointer. Shows your thought process.

⚠️ 10. Common Mistakes STEP 10 ▼
  1. Confusing "water at position i" with "total water"

    Water at i = min(leftMax, rightMax) - height[i]. Total = sum of all positions.

  2. Not including position i itself in leftMax/rightMax

    ❌ leftMax = max of elements strictly to the left → water can be negative

    ✅ leftMax includes height[i] itself → water is always ≥ 0

  3. Two pointer: moving the wrong pointer

    ✅ Always process and move the side with the SMALLER height. The taller side acts as a guaranteed wall.

🔑 11. When to Use This Pattern STEP 11 ▼

Signals to पहचान (identify):

  • ✅ "Trapping water" / "Container" / "Area between bars"
  • ✅ Each position depends on max from left AND max from right
  • ✅ Need to pre-compute prefix max and suffix max
  • ✅ Two pointer can replace prefix/suffix arrays when you only need the min of the two
🚫 12. When NOT to Use STEP 12 ▼
  • ❌ 2D trapping water (on a matrix) — needs BFS with priority queue
  • ❌ "Container With Most Water" (#14) — looks similar but logic is different (area between two lines, not filling gaps)
  • ❌ When bars can be moved/rearranged — this assumes fixed positions
💡 "Water at any point = min(tallest wall left, tallest wall right) - own height. Two pointers from ends: always process the shorter side — the taller side guarantees a wall."
