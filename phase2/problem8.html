#8 — Longest Streak of Active Users

🔖 Pattern: Sliding Window (Fixed Size)

🎯 Real-World Scenario START HERE ▼

Tum ek social media app ke analytics team mein ho. Tumhare paas last 10 days ka daily active user count hai:

[120, 95, 180, 200, 160, 140, 210, 190, 170, 150]

Boss bola: "Find the best 3-day streak — which consecutive 3 days had the maximum total active users?"

Possible 3-day windows:

  • [120, 95, 180] = 395
  • [95, 180, 200] = 475
  • [180, 200, 160] = 540
  • ... and so on

💡 Har window ke liye poora sum recalculate karna padega? Ya pichle window se reuse kar sakte hain?

🧠 1. Problem Understanding STEP 1 ▼

What exactly is being asked?

  • Given an array and a window size k, find the maximum sum of any k consecutive elements
  • This is the "Maximum Sum Subarray of Size K" problem

Input → Output:

  • Input: int[] nums, int k
  • Output: int — maximum sum of k consecutive elements

How is this different from Prefix Sum (#3)?

Aspect Prefix Sum (#3) Sliding Window (#8)
Query typeAny range (l, r)Fixed-size k consecutive
Pre-computationBuild prefix array firstNo pre-computation needed
SpaceO(n) for prefix arrayO(1) — just track window sum
Best forMultiple random queriesScanning all windows of size k

You COULD solve this with prefix sum, but sliding window is more elegant and uses O(1) space.

💪 2. Brute Force Thinking STEP 2 ▼

First instinct: For each starting position, sum k elements.

// Brute Force — O(n × k)
int maxSum = Integer.MIN_VALUE;
for (int i = 0; i <= n - k; i++) {
    int sum = 0;
    for (int j = i; j < i + k; j++) {
        sum += nums[j];
    }
    maxSum = Math.max(maxSum, sum);
}

Problem: O(n × k). If n = 100,000 and k = 50,000 → 5 billion operations.

The waste: Window [95, 180, 200] and window [180, 200, 160] share two elements (180, 200). We're re-adding them unnecessarily.

🤔 Jab window ek step right slide karti hai, sirf ek element naya aata hai aur ek purana jaata hai. Baaki same hain!

⚡ 3. Optimization Thinking STEP 3 ▼

Key Insight — The Train Window Analogy:

Imagine looking out of a train window. The window is fixed size (k seats wide).

  • As the train moves forward by 1 seat:
  • One new thing enters the window from the right
  • One old thing exits the window from the left
  • Everything in the middle stays the same

Translated to code:

// Window slides right by 1:
newSum = oldSum + nums[entering] - nums[leaving]

// Entering = right edge of new window
// Leaving  = left edge of old window

Instead of recalculating k additions, we do 1 addition + 1 subtraction = O(1) per slide.

💡 Pattern identified: Sliding Window (Fixed Size) — maintain a running sum, slide by add/remove

🎯 4. Final Approach (Pattern-Based) STEP 4 ▼

Pattern Name: Sliding Window — Fixed Size

Why this pattern fits:

  • We're looking at ALL contiguous subarrays of a fixed size k
  • Adjacent windows overlap by k-1 elements — massive reuse opportunity
  • Only need to track a running aggregate (sum, max, count, etc.)

Algorithm:

  1. Compute sum of first k elements (first window)
  2. Set maxSum = windowSum
  3. For i = k to n - 1:
    • windowSum += nums[i] - nums[i - k] (add new, remove old)
    • maxSum = Math.max(maxSum, windowSum)
  4. Return maxSum
🔍 5. Step-by-Step Dry Run STEP 5 ▼

Input: nums = [2, 1, 5, 1, 3, 2], k = 3

Step 1: First window

[2, 1, 5, 1, 3, 2] → windowSum = 2+1+5 = 8, maxSum = 8

Step 2: Slide window

Slide Window + New - Old windowSum maxSum
init [2,1,5,1,3,2] — — 8 8
1 [2,1,5,1,3,2] +1 -2 8+1-2 = 7 8
2 [2,1,5,1,3,2] +3 -1 7+3-1 = 9 9 ✓
3 [2,1,5,1,3,2] +2 -5 9+2-5 = 6 9

Result: maxSum = 9 (window [5, 1, 3])

4 windows checked with only 3 add/subtract operations (after initial sum). Brute force would do 4 × 3 = 12 additions.

⚙️ 6. Internal Mechanics STEP 6 ▼

Why sliding window is O(n) not O(n × k):

// Brute force: each window recalculates from scratch
Window 1: a + b + c           // 3 ops
Window 2:     b + c + d       // 3 ops (b,c recalculated!)
Window 3:         c + d + e   // 3 ops (c,d recalculated!)
Total: n × k operations

// Sliding window: reuse previous sum
Window 1: a + b + c           // k ops (only first time)
Window 2: prev + d - a        // 2 ops
Window 3: prev + e - b        // 2 ops
Total: k + 2(n-k) = O(n) operations

Sliding Window vs Two Pointer — what's the difference?

Aspect Two Pointer Sliding Window
FocusTwo specific elementsA range/subarray
MovementPointers move independentlyWindow slides as a unit (fixed) or expands/shrinks (variable)
TracksPair relationshipAggregate of window (sum, count, max)
ExamplePair sum, 3SumMax sum subarray, longest substring

Sliding window IS technically two pointers (left and right edge of window), but the mental model is different — you think in terms of "window" not "two elements".

📊 7. Complexity Analysis STEP 7 ▼
Time: O(n)
Space: O(1)
Approach Time Space n=100K, k=50K
Brute ForceO(n×k)O(1)5,000,000,000
Prefix SumO(n)O(n)100,000

Same time as prefix sum, but O(1) space. Sliding window wins for single-pass fixed-size problems.

☕ 8. Java Implementation STEP 8 ▼ 
public class MaxSumSubarrayK {

    public static int maxSumOfK(int[] nums, int k) {
        int windowSum = 0;

        // Build first window
        for (int i = 0; i < k; i++) {
            windowSum += nums[i];
        }

        int maxSum = windowSum;

        // Slide the window
        for (int i = k; i < nums.length; i++) {
            windowSum += nums[i] - nums[i - k];
            maxSum = Math.max(maxSum, windowSum);
        }

        return maxSum;
    }

    public static void main(String[] args) {
        int[] users = {120, 95, 180, 200, 160, 140, 210, 190, 170, 150};
        System.out.println(maxSumOfK(users, 3)); // 570 (210+190+170)
    }
}

💡 Notice how clean this is — no nested loops, no extra arrays. Just one variable tracking the window sum.

Variant: Return the average instead of sum

// Just divide by k at the end
public static double maxAvgOfK(int[] nums, int k) {
    // ... same sliding window logic ...
    return (double) maxSum / k;
}

⚠️ 10. Common Mistakes STEP 10 ▼
  1. Off-by-one in the sliding formula

    ❌ nums[i - k + 1] instead of nums[i - k]

    ✅ When i is the new right edge, the element leaving is at i - k

    Dry run with k=2: when i=2, leaving element is at index 0 = 2-2 ✓

  2. Forgetting to initialize maxSum with first window

    ❌ maxSum = 0 → wrong for all-negative arrays

    ✅ maxSum = windowSum (first window's sum)

  3. Not validating k <= n

    ✅ Add a guard: if (k > nums.length) return -1;

  4. Confusing fixed vs variable window

    Fixed window: size never changes, just slides. Variable window: size grows/shrinks based on a condition (that's Problem #9).

🔑 11. When to Use This Pattern STEP 11 ▼

Signals to पहचान (identify) this pattern:

  • ✅ "Maximum/minimum sum of k consecutive elements"
  • ✅ "Average of subarrays of size k"
  • ✅ "Find all subarrays of size k with property X"
  • ✅ Problem mentions "consecutive", "contiguous", "subarray" + fixed size
  • ✅ "Sliding window maximum" (needs deque — advanced variant)

Fixed window template:

// Generic fixed-size sliding window
// 1. Build first window of size k
// 2. For each new element entering:
//    a. Add new element to window state
//    b. Remove oldest element from window state
//    c. Update answer
🚫 12. When NOT to Use STEP 12 ▼
  • ❌ Window size is NOT fixed — use variable-size sliding window (Problem #9)
  • ❌ Elements are not contiguous (e.g., "pick any k elements") — that's a different problem (sorting/heap)
  • ❌ You need the actual subarray, not just the aggregate — need to track start index too
  • ❌ "Maximum subarray sum" without fixed k — that's Kadane's algorithm (different pattern)
💡 "Fixed-size window sliding karti hai toh sirf ek element aata hai, ek jaata hai. Poora recalculate mat karo — add/remove karo."
🌉 14. Bridge to Next Problem NEXT ▼

Ab tum fixed-size window slide kar sakte ho in O(n).

But what if window size fixed nahi hai? "Find the smallest subarray whose sum is ≥ target" — window size depends on the data.

  • Window chhoti hai aur sum target se kam? → Expand (right badhao)
  • Sum target se zyada ho gaya? → Shrink (left badhao)

That's the Variable-Size Sliding Window — Problem #9: Shortest Refueling Stop on Highway.

→ Fixed window (size given) → Variable window (size depends on condition) — harder but more powerful!
