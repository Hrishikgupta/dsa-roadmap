#6 — Pair of Items That Hit Target Budget

🔖 Pattern: Two Pointer (Pair Sum on Sorted Array)

🎯 Real-World Scenario START HERE ▼

Tum ek office supplies store mein ho. Boss ne ₹9 ka budget diya hai aur bola:

"Exactly do items kharido jinki total cost ₹9 ho."

Available items (sorted by price): [1, 2, 3, 4, 5, 6, 7, 8]

Possible pairs: (1,8), (2,7), (3,6), (4,5) — all sum to 9.

But tumhe sirf ek pair return karna hai — aur efficiently.

💡 Array sorted hai — kya hum bina har pair check kiye answer nikal sakte hain?

🧠 1. Problem Understanding STEP 1 ▼

What exactly is being asked?

  • Given a sorted array and a target sum, find two numbers that add up to the target
  • Return their indices (1-indexed in LeetCode's "Two Sum II")
  • Exactly one solution is guaranteed

Input → Output:

  • Input: int[] numbers (sorted), int target
  • Output: int[] — indices of the two numbers

Two versions of this problem:

Version Input Best Approach
Two Sum (unsorted) — LC #1Unsorted arrayHashMap (Problem #2 pattern)
Two Sum II (sorted) — LC #167Sorted arrayTwo Pointer (this problem)

Connection to Problem #5:

  • #5 used two pointers on two separate arrays, both moving forward
  • #6 uses two pointers on one array, moving from opposite ends inward
💪 2. Brute Force Thinking STEP 2 ▼

Approach 1: Check every pair — O(n²)

for (int i = 0; i < n; i++) {
    for (int j = i + 1; j < n; j++) {
        if (nums[i] + nums[j] == target) return new int[]{i, j};
    }
}

Works but O(n²). Doesn't use the sorted property at all.

Approach 2: HashMap — O(n) time, O(n) space

HashMap<Integer, Integer> map = new HashMap<>();
for (int i = 0; i < n; i++) {
    int complement = target - nums[i];
    if (map.containsKey(complement)) return new int[]{map.get(complement), i};
    map.put(nums[i], i);
}

O(n) time but O(n) space. Still doesn't use sorted property.

🤔 Array sorted hai — kya hum O(n) time AND O(1) space mein kar sakte hain?

⚡ 3. Optimization Thinking STEP 3 ▼

Key Insight — The Weighing Scale Analogy:

Imagine a weighing scale. You need exactly 9 kg. You have sorted weights: [1,2,3,4,5,6,7,8].

  • Start with the lightest (1) and heaviest (8): 1 + 8 = 9 ✓ Done!
  • But what if target was 11? 1 + 8 = 9 → too small → need heavier left side → move left pointer right
  • What if target was 5? 1 + 8 = 9 → too big → need lighter right side → move right pointer left

The logic:

  • sum == target → found it!
  • sum < target → sum chhota hai, bada karna hai → left pointer aage badhao (bigger number)
  • sum > target → sum bada hai, chhota karna hai → right pointer peeche lao (smaller number)

Why this works:

  • Array sorted hai, so left pointer badhane se sum BADHEGA
  • Right pointer ghatane se sum GHATEGA
  • Har step mein hum ek element permanently eliminate kar rahe hain
  • At most n steps → O(n)

💡 Pattern identified: Two Pointer — opposite ends, shrink inward based on sum comparison

🎯 4. Final Approach (Pattern-Based) STEP 4 ▼

Pattern Name: Two Pointer — Opposite Ends (Pair Sum)

Pre-requisite: Array MUST be sorted. If not, sort first O(n log n) or use HashMap.

Algorithm:

  1. left = 0, right = n - 1
  2. While left < right:
    • sum = nums[left] + nums[right]
    • If sum == target → return {left, right}
    • If sum < target → left++
    • If sum > target → right--
  3. No pair found → return empty (shouldn't happen if guaranteed)
🔍 5. Step-by-Step Dry Run STEP 5 ▼

Input: nums = [2, 3, 5, 8, 11, 15], target = 13

Step left right nums[L] nums[R] Sum vs Target Action
1 0 5 2 15 17 17 > 13 right-- → 4
2 0 4 2 11 13 13 == 13 ✓ FOUND! return {0, 4}

Result: indices [0, 4] → values (2, 11) → sum = 13 ✓

Only 2 steps to find the answer in a 6-element array. Brute force would check up to 15 pairs.

Another example — target = 10:

⚙️ 6. Internal Mechanics STEP 6 ▼

Why we never miss the answer:

Proof by elimination — when we move a pointer, we're safely discarding impossible pairs:

  • When sum > target and we do right--:
    • We're saying: "nums[right] paired with ANY remaining left element will be too big"
    • Why? Because left can only move right (bigger), making sum even bigger
    • So nums[right] can never be part of the answer → safe to eliminate
  • When sum < target and we do left++:
    • We're saying: "nums[left] paired with ANY remaining right element will be too small"
    • Why? Because right can only move left (smaller), making sum even smaller
    • So nums[left] can never be part of the answer → safe to eliminate

Each step eliminates exactly one element → at most n steps → O(n).

Comparison of all three approaches:

Approach Time Space Needs Sorted?
Brute ForceO(n²)O(1)No
HashMapO(n)O(n)No
Two PointerO(n)O(1)Yes

Two Pointer wins when input is sorted. HashMap wins when it's not (and you can't sort).

📊 7. Complexity Analysis STEP 7 ▼
Time: O(n)
Space: O(1)

Each pointer moves at most n times total. No extra data structures. Best possible for this problem.

If array is unsorted and you sort first: O(n log n) + O(n) = O(n log n). Still better than O(n²) brute force.

☕ 8. Java Implementation STEP 8 ▼

Two Pointer (sorted array):

public class TwoSumSorted {

    public static int[] twoSum(int[] nums, int target) {
        int left = 0, right = nums.length - 1;

        while (left < right) {
            int sum = nums[left] + nums[right];
            if (sum == target) {
                return new int[]{left, right};
            } else if (sum < target) {
                left++;
            } else {
                right--;
            }
        }
        return new int[]{}; // no pair found
    }

    public static void main(String[] args) {
        int[] nums = {2, 3, 5, 8, 11, 15};
        int[] result = twoSum(nums, 13);
        System.out.println(result[0] + ", " + result[1]); // 0, 4
    }
}

Bonus: HashMap approach (unsorted array — LC #1):

public static int[] twoSumUnsorted(int[] nums, int target) {
    HashMap<Integer, Integer> map = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int complement = target - nums[i];
        if (map.containsKey(complement)) {
            return new int[]{map.get(complement), i};
        }
        map.put(nums[i], i);
    }
    return new int[]{};
}

💡 Interview tip: Always ask "Is the array sorted?" — it changes the optimal approach entirely.

⚠️ 10. Common Mistakes STEP 10 ▼
  1. Using two pointers on an unsorted array

    ❌ Two pointer pair sum ONLY works on sorted arrays

    ✅ If unsorted → use HashMap approach or sort first

  2. Using left <= right instead of left < right

    ❌ When left == right, you're using the same element twice

    ✅ Strict < ensures two different elements

  3. Integer overflow on sum

    ❌ nums[left] + nums[right] can overflow if values are near Integer.MAX_VALUE

    ✅ Use long sum = (long)nums[left] + nums[right] for safety

  4. Returning 1-indexed vs 0-indexed

    LeetCode #167 (Two Sum II) expects 1-indexed. Read the problem carefully.

🔑 11. When to Use This Pattern STEP 11 ▼

Signals to पहचान (identify) this pattern:

  • ✅ "Find pair with given sum" on sorted array
  • ✅ "Two numbers that add to target"
  • ✅ Any problem where you need to find two elements satisfying a condition on a sorted array
  • ✅ "Pair with minimum/maximum difference" on sorted array
  • ✅ Problem says "sorted" + "pair" → two pointer is almost always the answer

Decision tree:

Find pair with sum = target?
├── Array sorted? → Two Pointer O(n), O(1)
├── Array unsorted, can sort? → Sort + Two Pointer O(n log n)
└── Need original indices? → HashMap O(n), O(n)
🚫 12. When NOT to Use STEP 12 ▼
  • ❌ Array is unsorted and you need original indices (sorting destroys index info → use HashMap)
  • ❌ Finding ALL pairs (not just one) — two pointer finds one, need modification for all
  • ❌ Three numbers (3Sum) — need two pointer inside a loop (that's Problem #7!)
  • ❌ Target is a product/XOR instead of sum — different math, different approach
💡 "Sorted array + pair sum = Two Pointers from opposite ends. Sum bada? Right ghatao. Sum chhota? Left badhao."
